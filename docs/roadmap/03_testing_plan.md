## Порядок построения (в рамках I7)

1. **I1 сначала** — разгитигнорить `tests/`, иначе CI не увидит тесты.
2. Ввести pytest как канонический раннер (сейчас часть тестов = запускаемые скрипты) + `conftest`.
3. Coverage-замер, порог в CI (стартовый, поднимать).
4. Закрыть reactions/outbound-security gaps (F5, outbound-слой) — самые опасные пустоты.
5. Каждый продукт-воркстрим (P#) приходит со своими unit+contract+honest-stub-тестами.

## Особенности, которые нельзя потерять

- **Симуляции — часть безопасности, не «e2e для галочки»**: они проверяют, что файрвол реально блокирует, а не «театр» (память: injection-театр уже снимали).
- **Reactions-as-contract**: коды реакций — публичный контракт для Claude; тест на каждый код + recovery.
- **Outbound = сервер атакует клиента**: уникальная для MCP поверхность; вывод инструмента как вектор prompt-injection обязателен в покрытии.

## Эмуляция реальной работы структуры (E-матрица)

> Цель: проверить не «код работает», а **реальные сценарии ИИ** с `structure_create`/`structure_link`/`structure_migrate`/`structure_status`/`structure_check_integrity` — включая **циклы рекомендаций сервера** (ORPHAN / childless) и **сжатие проходов**. Все E-тесты — против ЖИВОГО сервера (JSON-RPC на `/mcp`), ассерт по `ToolResult.facts` + `data.orphan_notices` + кодам реакций, изоляция через свежий workspace на тест. Связано с **F25** (слепое пятно ФС↔реестр) и памятью `structure-templates-feature`.
>
> Иерархия (из шаблонов `config/templates/workspace/*.tpl.yaml`): `niche → network(сетка) → { channel→video | competitor_channel→competitor_video }`. Конкурент группируется по НАШЕМУ каналу (`competitors/{наш_канал}/`), и `REQUIRED_PARENT_TYPE={competitor_channel: channel}` → конкурент без нашего канала = **ORPHAN**.
> Depth-control (template_engine): материализуются свои folders/files + контейнеры детей; в детей спускаемся ТОЛЬКО для НАЗВАННЫХ (`children={type:[имена]}`), остальные → `deferred_children`/`ChildDeferred`. `children` прокидывается рекурсивно во все уровни (`template_engine.py:202`) — это и есть «сервер подставляет имя детям» и механизм сжатия проходов.

### E-A — Конфиг-матрица ниши (цель: циклы рекомендаций)

Создание ниши в 4 конфигурациях; ассерт, что сервер выдаёт ПРАВИЛЬНУЮ рекомендацию:

| ID | Конфигурация (named children) | Ожидаемые facts | Ожидаемая рекомендация сервера |
|---|---|---|---|
| E-A1 | ниша, без сетки/канала/конкурента (**холодный старт**) | `NodeCreated(niche)`, `ChildDeferred(network)` | **cold-start advisory** (🔴 НЕ реализовано → xfail-spec, F26): сервер обязан уведомить, что объектов ещё нет → данных для анализа нет; **рекомендованный путь** — изучить конкурентов (прислать скриншоты каналов конкурентов + 2-3 транскрибации сценариев); **fallback-путь** (клиент не владеет конкурентами) — начать с личного канала, а при появлении конкурента сервер сам свяжет конкурента↔личный канал и фоном физически переместит ОБА в правильную иерархию |
| E-A2 | ниша+сетка+**канал**, без конкурента | `NodeCreated`×3, `ChildDeferred(video, competitor_channel)` | **childless** (`structure_status.find_childless(channel, competitor_channel)`) — мягко: у канала нет конкурента |
| E-A3 | ниша+сетка+**конкурент**, без нашего канала | `NodeCreated(…competitor_channel)`, `EntityOrphaned` | **ORPHAN** — конкурент без нашего канала; recovery = создать/связать наш канал (`structure_link`/`structure_migrate`) |
| E-A4 | ниша+сетка+**канал+конкурент** | `NodeCreated`×4 | **KEY**: пришпилить фактическое поведение — конкурент named-child сетки получает `parent_ids=[…network]`, НЕ channel → по коду **всё ещё ORPHAN** до `structure_link`. Тест фиксирует, авто-подстановка `{наш_канал}` (§4) реализована или нет. Дыра = находка |

### E-B — 4 точки входа × depth-control

`structure_create` можно начать с любого уровня; сервер подставляет имена названным детям, остальных откладывает. Для каждой точки: (a) голое создание → все дети deferred; (b) с одним названным ребёнком → его поддерево материализовано, соседи deferred.

| ID | Точка входа | (a) голое | (b) с названным ребёнком |
|---|---|---|---|
| E-B1 | `niche` | `deferred_children=[network]` | `children={network:[N]}` → сетка материализована, `children[0].deferred_children=[channel,competitor_channel]` |
| E-B2 | `network` (сетка) | `deferred=[channel, competitor_channel]` | `children={channel:[C]}` → канал есть, конкурент deferred |
| E-B3 | `channel` | `deferred=[video]`, `tables_pending`=channel_data/config | `children={video:[V]}` → видео-поддерево материализовано |
| E-B4 | `competitor_channel` | `deferred=[competitor_video]` | `children={competitor_video:[CV]}` → видео конкурента материализовано |

Ассерты для всех: `tables_pending` для `kind:table` (не материализованы, G16); валидное имя = один сегмент (кривое имя → `VALIDATION_ERROR`, соседи создаются — пофрагментный скип §1).

### E-C — Проходы: 3 / 2 / 1 (метаморфный инвариант)

Ключевая проверка: **для фиксированного набора знаний ИИ итоговое состояние (реестр + файлы + notices) ИДЕНТИЧНО, сколько бы проходов ни ушло** — проходы отличаются только промежуточными `ToolResult`. Три набора знаний, каждый — на уровне КАНАЛА и уровне СЕТКИ:

| ID | Что ИИ знает заранее | 3-прохода | 2-прохода (сжатие) | Инвариант / рекомендация |
|---|---|---|---|---|
| E-C1 | ниша + наш канал + канал конкурента | niche → channel → competitor+`link` | niche(`children={network,channel}`) → competitor+`link` | итог = полностью связано, **0 orphan**; 3≡2≡1-pass end-state |
| E-C2 | ниша + конкурент, но НЕ наш канал | niche → competitor (orphan) → позже канал → `link`+`migrate` | niche(`children={network,competitor_channel}`) | конкурент **ORPHAN**; при появлении канала — сервер связывает и **фоном физически переносит ОБА** в `competitors/{наш_канал}/` (сейчас ручной `structure_migrate` по одной сущности — F27, авто-оркестрации нет → часть ассертов = xfail-spec) |
| E-C3 | ниша + наш канал, но НЕ конкурент | niche → channel | niche(`children={network,channel}`) | канал **childless**; сервер мягко рекомендует добавить конкурента |
| E-C1′..C3′ | то же на уровне **сетки** (знает сетку + …) | вход с `network` | `network(children={…})` | те же инварианты, точка входа = сетка |

Метаморфное правило: `end_state(3-pass) == end_state(2-pass) == end_state(1-pass)` по `structure_check_integrity` (total_entities, by_type, issues) для одного набора знаний. Расхождение = баг сжатия/подстановки.

### E-D — Деструктив над таблицами (реакция сервера + доводка рекомендаций)

Цель: физически ломать таблицы-детей (данные в столбцах, сами столбцы, целые листы, перенос данных под формулами) и проверять РЕАКЦИЮ сервера — блокирует / предупреждает / рекомендует восстановление. Где реакции нет или она слабая → находка + попутно дорабатываем рекомендацию. Два слоя защиты: `core/tables` (данные) и `core/excel` (форма).

| ID | Деструктив | Инструмент | Ожидаемая реакция |
|---|---|---|---|
| E-D1 | запись в computed-столбец (формула) | `table_set` | `COMPUTED_READONLY` (немедленный отказ ДО очереди) + recovery «меняй исходные столбцы» |
| E-D2 | значение вне enum | `table_set` | `ENUM_VIOLATION` + recovery «одно из enum схемы» |
| E-D3 | перезапись существующей формулы | `excel_insert_formula` (overwrite=false) | `FORMULA_PROTECTED` + recovery «overwrite=true осознанно» |
| E-D4 | удалить последний лист книги | `excel_delete_sheet` | `LAST_SHEET` (guard) |
| E-D5 | удалить столбец, на который ссылается формула | `excel_delete_column` | 🔴 **GAP (F28)**: молча ломает, реакции в момент нет; `validate_formulas` ловит лишь cached-токены → ассерт-факт коррапта = находка, «должная» реакция = `xfail`-spec |
| E-D6 | переместить столбец под формулой (сдвиг данных → ломает формулу) | `excel_move_column` | 🔴 **GAP (F28/F29)**: молча корраптит валидной-но-неверной ссылкой, НЕ `#REF!`, `validate_formulas` слеп → находка + `xfail`-spec на должную реакцию |
| E-D7 | удалить данные ячеек / строку | `table_delete` | пришпилить фактическую реакцию (queue/мягко?) — доводим рекомендацию |
| E-D8 | удалить всю таблицу-ребёнка (файл) | `table_delete`/`fs_delete` | реакция + **связка с F25**: `structure_check_integrity` должен показать `broken_reference`/сироту после удаления зарегистрированного файла |

### E-E — Тайминг присвоения ID и порядок создания детей

Цель: проверить систему автогенерации ID и КОГДА присваивается ID относительно таблиц. Инвариант владельца: **ID присваивается ДО таблиц**, но возможен сценарий, когда часть детей по разным причинам создаётся ПОСЛЕ таблиц — генерация обязана остаться консистентной.

| ID | Проверка | Ассерт |
|---|---|---|
| E-E1 | node_id присвоен при создании узла, до материализации таблиц | facts `NodeCreated.id` присутствует; `create_node` даёт `node_id` (template_engine:129) раньше, чем строятся .xlsx |
| E-E2 | таблица-ребёнок получает `file_id` в момент ОТКЛАДЫВания (Ф3 не построена) | `tables_pending[].file_id` есть, .xlsx НЕ создан (honest-stub G16); реестр знает `table_file` (server.py:553) |
| E-E3 | уникальность ID — нет коллизий | повторные создания → нет `DUPLICATE_ID`; `register(check_unique=True)` даёт `DUPLICATE_ID` при занятом ID др. типом |
| E-E4 | **out-of-order**: ребёнок создан ПОСЛЕ таблиц (видео названо позже / отложенный ребёнок материализован после фазы таблиц) | ID нового ребёнка уникален, `parent_ids` корректны, ранее выданный `file_id` таблицы валиден/связан — порядок не рвёт целостность |
| E-E5 | запись в таблицу идёт через очередь (write.json), валидация ДО очереди | `table_append`/`table_set` кладут в очередь; computed/enum проверены до постановки (tables_core:129) |

### E-F — Устойчивость формул к неполным данным (config/флаги, F30)

Цель: формулы таблиц НЕ ломаются при частичных/отсутствующих данных; устойчивость управляется **декларативно** (флаги в `channel_config`), не хардкодом. Полных данных не бывает — надо проверять деградацию. Механизмы из спеки: `scene_profile.enabled` (гасит тип в формуле уникальности — «тихий столбец»), `niche_weight` (вес), `automation_rules.condition` (пороги), `signal_on_reuse/reuse_threshold` (сигналы). Все — `xfail`-spec до Ф3 (loader/фаза формул не построены).

| ID | Неполнота данных | Механизм устойчивости | Ассерт |
|---|---|---|---|
| E-F1 | тип фрагмента отсутствует/выключен | `scene_profile.enabled=false` → формула уникальности пропускает тип | `uniqueness_score` считается без типа, **НЕ** `#DIV/0!`; столбец остаётся («тихий») |
| E-F2 | нет части асетов (svg/scenes/audio) или их вариаций | `niche_weight` по присутствующим + guard | результат валиден, не `#REF!`/`#N/A` |
| E-F3 | нет данных конкурентов | competitor-зависимые формулы деградируют | напр. `prediction_accuracy=PENDING`, не ошибка |
| E-F4 | частичные данные нашего канала | формула по доступным столбцам | нет каскада ошибок вниз по зависимостям |
| E-F5 | глобальный асет **+ его вариации** (`AST_x`) | уникальность учитывает вариации; `reuse>threshold` → сигнал | `signal_on_reuse`/`reuse_threshold` срабатывает; сервер рекомендует добавить вариацию («приелся») |
| E-F6 | рекомендации по уникальности | `automation_rules.condition` (`uniqueness_score<0.60`) | сервер алертит/блокирует по `severity`; рекомендация уникальности асетов и др. данных |

Правило (anti-hardcode): устойчивость и веса — из конфига (`scene_profile`/`niche_weight`/`automation_rules`), НЕ `if type missing` / магические пороги в коде. Тест ловит и хардкод, и поломку формулы.

## Eval-слой: качество данных, асетов и поиска (сверх контракта, F31)

> Отдельный КЛАСС проверок: не «инструмент вернул валидный `ToolResult`» (это E-A…E-F/контракт), а **насколько ХОРОШ результат** — качество выбора информации, поток данных, релевантность поиска. Forward к продукту (P1–P7 = стабы), НО поисковый слой (`core/search`) существует и тестируем сейчас.
>
> **Экономия токенов (принцип владельца):** данные для eval — синтетические/вымышленные + локальные модели + загрузка из интернета, НЕ дорогие облачные LLM. В CI — детерминированные фикстуры, провайдеры замоканы (правило test-master: реальные API провайдеров в юнит-тестах запрещены). Реализм через локальные модели/загрузку — отдельная opt-in eval-прогонка, не блокирует CI.

### E-H — Качество пайплайна (eval: выбор контекста)

Оцениваем КАЧЕСТВО выбора информации для сценария / аудио / картинок. Ключ — разделить ДВА класса провала:

| Класс провала | Что значит | Как ловим |
|---|---|---|
| **промпт хромает** | плохая инструкция модели | held-out варианты промпта при ФИКС. контексте |
| **поток данных плох** | лишний/ненужный контекст ЛИБО недостаток | метаморфно варьируем контекст при ФИКС. промпте |

Метод: metamorphic (контекст + шум → качество не должно рушиться; контекст − нужное → падает предсказуемо) + judge-оценка. **Разделение промпт ↔ данные — главный диагностический выход** (чинить то, что реально виновато).

### E-I — Умный поиск (`core/search`) — тест и прокачка

`FsSearcher`+`QueryPlanner`, инструменты `search_quick`/`search_multi`/`search_tables` по таблицам+ФС. Сейчас `tests/quick/test_search.py` 24/24, но не ревьюен (F4), без relevance-eval.

| ID | Проверка |
|---|---|
| E-I1 | релевантность/точность: нужное найдено, мусор не поднят (precision/recall на фикстурах E-G) |
| E-I2 | две поверхности — поиск по таблицам vs ФС — обе корректны |
| E-I3 | adversarial/outbound: вывод поиска как вектор prompt-injection (сервер-как-атака) |
| E-I4 | поиск ПИТАЕТ E-H: плохой поиск = плохой поток данных → замыкание на качество контекста |

## Агентный adversarial-харнесс (симуляции как многовариантный движок, F32)

> Симуляции (`bot_army`/`virus_injection`/`cache_*`) — штатный движок СЛОЖНЫХ многовариантных тестов, а не «e2e для галочки». Два режима: (1) физическая симуляция ситуаций через консоль/пакеты, (2) **комбинирование через агентов** — эмерджентные и даже абсурдные сценарии. Расширяем существующий паттерн (`tests/<name>/{test,history}` + `scenarios.yaml`) до роя агентов. Это красная команда ПРОТИВ своего сервера (авторизованный тест своих защит) — цель доказать, что сервер нельзя пробить (inbound) и нельзя превратить в оружие против клиента (outbound).

### Роли агентов (пример: 10 честных + 10 злоумышленников в одном прогоне)

| Роль | Модели поведения | Что проверяет |
|---|---|---|
| **Честный клиент** ×N | новичок / power-user / хаотичный / конкурент-центричный / пакетный-писатель | работа НЕ ломается ложными срабатываниями (нет false-positive бана); легитимный поток проходит под нагрузкой |
| **Злоумышленник inbound** ×N | пакетные атаки, path-traversal (D1/D29), injection, rate-limit-bypass, cache-переполнение, вирус-payload | `core/firewall` реально банит/блокирует И подключён (регрессия D2/D8) |
| **Злоумышленник outbound** ×N | сервер-как-атака через Claude AI Web: prompt-injection в выводе инструмента, обход защиты Claude паттернами, кража/эксфильтрация данных, `fs_delete/write/move` по файлам пользователя и КЛИЕНТСКИМ данным, генерация вредоносного ПО | outbound-защита держит: деструктив под gate, нет эксфильтрации, сервер не ассистирует малвари/обходу |

### Критерии успеха (одновременно)

1. **Inbound держит:** каждый вектор атакующего → блок/бан с кодом реакции; дыра в защите = 🔴 находка + регрессия.
2. **Outbound держит:** ни один агент не смог через сервер украсть данные, удалить пользовательские/клиентские файлы, сгенерировать малварь или обойти защиту Claude — вывод инструмента санитизирован.
3. **Честные не страдают:** смесь не даёт ложных банов легитимным клиентам (баланс безопасность↔доступность; связка с G18/rate-limiter D14/D16).
4. **Эмерджентность:** комбинация ролей вскрывает то, что одиночный поток не видит (гонки состояния firewall, обход через чередование honest/attacker с одного IP).
