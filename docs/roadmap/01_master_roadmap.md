## Целевая структура (закон размещения) — конформность

**Авторитет:** `spec/instructions/ИНСТРУКЦИЯ_структура_и_ядро.md` — «карта и закон» (§0 «у каждого файла ровно
один дом», §2 скелет, §5 карта артефактов). README показывает **факт**, этот раздел — **цель**, куда ведут
воркстримы. Дерево не дублируем — оно в законе; здесь маппинг «целевой каталог → воркстрим».

| Целевой каталог (закон §2/§5) | Правило | Статус | Воркстрим |
|---|---|---|---|
| `tools/<group>/` — тонкие обёртки, **свой каталог на каждую группу** (`filesystem`, `tables`, `excel_engine`, `media`, `video`) + `<group>_core` | инструменты только здесь; логика в `*_core`, наружу тонкий контракт | 🔲 пусто (логика в `server.py` монолит) | **A2** (распил монолита по группам) |
| `core/providers/<provider>/` — **каталог на каждого провайдера** (`stt`, `tts`, `img`, `ffmpeg` + `ffmpeg/wrappers/`) | способность = адаптер за единым интерфейсом | 🟠 каталоги есть, адаптеры = заглушки | **P1–P4** + P-video |
| `tools/media/` (`tts_*`/`stt_*`/`img_*`) · `tools/video/` (монтаж) | тонкие обёртки над `core/providers` | 🔲 | **P4′** (media), **P7** (video) |
| `pipeline/entry_points/` (4 точки = наборы предусловий) + `pipeline/steps/` (единая библиотека шагов) | входы не копируют воркфлоу; шаги переиспользуются | 🔲 пусто | **P5** (steps), **P6** (entry_points) |
| `scripts/introspect_tables.py` — утилита вне рантайма | только `scripts/`, не `core`/`pipeline` | 🔲 | **A1′** |
| `docs/dev/tools/<X>` — **зеркалит** `tools/<X>`/`core/providers/<X>` (простое→1 файл, сложное→папка+overview) | доки рядом-параллельно коду | 🔲 | **I8** (docs) |
| `config/ops/<cat>.ops.yaml`, `config/model_routing.yaml`, `config/paths.yaml` | операции/маршрутизация/пути — декларации | ⚠️ **СПОРНО (F23)** — «закон» требует, media-план упраздняет | **ждёт решения владельца** |

**Закон размещения способностей (§5, обобщение для будущих):** любая новая способность = адаптер
`core/providers/<cap>/` + тонкие инструменты `tools/<category>/` + операции `config/ops/<category>.ops.yaml`
(+ данные в `workspace/`). Все воркстримы P/A обязаны следовать этому шаблону — иначе файл «окажется где попало».

> **F23 (конфликт спек, ждёт владельца):** `ИНСТРУКЦИЯ_структура_и_ядро.md` (закон) требует `config/ops/*` +
> `model_routing.yaml`; `media_tools_deployment.md §0` их упраздняет (консолидация в `channel_config.resource_limits`).
> От ответа зависят **A1/A3** и наличие `config/ops` как цели. До решения — A2/провайдеры/pipeline планируем как есть
> (эта часть структуры в обоих документах одинакова).

## Фазовая раскладка (по зависимостям и ценности)

- **Фаза 0 — Фундамент (разблокирует всё):** `I1` → `I2` → `I4` → `A4` → `I3`.
  VCS-гигиена, packaging, типы/линт, config-правда, CI. Без этого любой продукт-код кладётся на песок.
- **Фаза 1 — Архитектура-правда:** `A1` → `A2` → `A3` → `A6`.
  Декларативный слой + распил монолита + реакции. Закрывает «README врёт».
- **Фаза 2 — Продукт (то, ради чего проект):** `P3` → `P2`/`P4` → `P1` → `P5` → `P6` → `P7`.
  Провайдеры (от простого локального STT к внешнему FFmpeg) → шаги → оркестрация → видео.
- **Фаза 3 — Зрелость:** `I5` (observability) → `I6` (auth/secrets) → `A5` (search-харденинг) → `I8` (доки+аудит).
- **Сквозное:** `I7` тесты — каждый воркстрим приходит со своими тестами (см. план тестов).

## Оценка объёма

~21 воркстрим × 1–2 сессии ≈ **20–30 сессий**. Совпадает с ориентиром «10–20+, сколько понадобится».
Приоритет фазы 1 (что делать вперёд) — **все три оси в работе**, но стартуем с Фазы 0, т.к. она разблокирует безопасное исполнение остальных (тесты в git + CI ловят регресс продукт-кода).

## Definition of "Enterprise-ready"

Проект считается доведённым, когда: продукт (P1–P7) работает e2e; архитектура соответствует README (A1–A6); есть CI/типы/логи/auth/packaging (I1–I6); тестовое покрытие по плану (I7); доки публикуемы (I8); аудит-словарь имеет артефакты (X1).
