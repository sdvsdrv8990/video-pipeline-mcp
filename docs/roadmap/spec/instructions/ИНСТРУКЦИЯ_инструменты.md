# ИНСТРУКЦИЯ: Инструменты сервера

> Это спека интерфейса инструментов. Внутренности слоя данных (row-by-id, RMW,
> очередь-на-таблицу) — в **Брифе табличных инструментов**, поведение при сбоях —
> в **server_reactions.yaml**, размещение файлов — в **ИНСТРУКЦИИ структуры**.
> Здесь — что за инструменты, как они устроены и где граница между ними. Без дублирования.

---

## 0. ЗАЧЕМ РУЧНЫЕ ИНСТРУМЕНТЫ: гибкость

Шаблон — быстрый путь (готовая структура одним вызовом). Ручные инструменты —
путь под себя:

- **Подправить шаблон** — взять стандартную таблицу и добавить/убрать столбец, лист.
- **Собрать своё** — построить таблицу с нуля ручными вызовами `excel_*`.

Оба пути дают одно: **валидную таблицу, которую понимает слой данных.**

> **ЗАКОН СИНХРОНИЗАЦИИ.** Любая ручная правка *структуры* (столбец, лист,
> формула) обязана обновить `schema.json` этой таблицы. Иначе слой данных не
> знает про новый столбец и не защитит его (не поймёт `writable`/`computed`).
> После ручных изменений → перегенерация схемы интроспектором
> (`scripts/introspect_tables.py`). Гибкость работает ПОВЕРХ контракта, не в обход.

## 1. СКЕЛЕТ ЛЮБОГО ИНСТРУМЕНТА (един для всех категорий)

```
ИНТЕРФЕЙС   тонкая обёртка с Pydantic-контрактом   ← это видит Claude (5–10 строк)
    ↓
ДВИЖОК      generic-ядро категории (*_core)        ← вся логика, один на категорию
    ↓
КОНФИГ      реестр операций (config/ops/*.ops.yaml) ← добавить операцию = строка в YAML
```

Универсальные правила (нарушать нельзя):

- **Инструмент тупой и чистый.** Не помнит состояние между вызовами, не знает,
  что идёт после. Факт о сделанном **возвращает** (`ToolResult.facts`), а пишет
  в state — фреймворк. Состояние процесса — в `core/state/`, не в инструменте.
- **Контракт строгий.** Своя Pydantic-схема на инструмент; обязательные поля
  действительно обязательны. Никаких «опционально, зависит от operation»
  (router-каша запрещена). Где режимов несколько — явный дискриминатор `mode`.
- **Предусловия — guards в `ops.yaml`,** не `if` в коде. Missing Component
  Protocol живёт как guard, а не как ветка внутри инструмента.
- **Единый конверт ответа** `ToolResult{status, data, error, facts}` — одинаковый
  у всех инструментов, чтобы Claude видел один формат от любого из сотен.

**Правило слияния.** Объединять можно только операции с одинаковой формой входа
И одинаковой валидацией. Но ясность важнее счётчика: `fs_move` и `fs_rename`
сливать в `fs_relocate` не стоит — самодокументирующиеся имена помогают Claude
выбрать намерение без ошибки. Меньше инструментов — не цель. Ясность — цель.

## 2. КАТЕГОРИЯ 1 — Файловая система (скелет рабочего пространства)

Тонкие `fs_*` + `filesystem_core` + `config/ops/filesystem.ops.yaml`.

| Инструмент | Назначение |
|---|---|
| `fs_create_project_structure` | материализует структуру по шаблону ИЛИ один недостающий фрагмент |
| `fs_create_file` | создать файл (.md/.json/.yaml/.py/.txt) с начальным содержимым |
| `fs_create_python_script` | .py с базовым каркасом (imports/main/error handling) |
| `fs_read_file` | прочитать файл (с лимитом размера) |
| `fs_write_file` | полная перезапись (осторожно; для данных — очередь, не это) |
| `fs_move` / `fs_rename` | переместить / переименовать (раздельны ради ясности) |
| `fs_delete` | удаление (требует подтверждения или `force=true`) |
| `fs_smart_search` | поиск по directory/extension/keyword (замена векторного поиска) |
| `fs_get_directory_tree` | дерево каталога с типами файлов |

Ключевое для `fs_create_project_structure`:

- **Дискриминатор режима** `mode: "template" | "single"` — Claude явно говорит,
  что делает; Pydantic по `mode` валидирует нужный набор полей.
- **Имя + ID на каждом фрагменте.** Имя — для человека/поиска; ID — для связей
  (ДОГМА 2). **ID присваивает сервер** (Genesis), он приходит в `facts`, не на входе.
- **На входе — только `parent_ids`** (куда крепим). Родительские ID опциональны
  по уровню вложенности: нет сетки (`network_id=null`), но есть канал — канал
  создаётся. Guard `parent_exists` отклоняет ссылку на несуществующего родителя.
- **Пофрагментная валидация имени.** Нет валидного имени → фрагмент пропускается
  (`skip`), но соседи с именами создаются. Это НЕ атомарный откат. Пропуск
  возвращается Claude как факт (чтобы он знал про неполноту), код `STRUCTURE_INCOMPLETE`.

## 3. КАТЕГОРИЯ 2 — Структура таблиц (Excel Engine) = ручная гибкость

Это и есть ручное управление таблицами. Полный CRUD над **структурой** (скелетом).

| Инструмент | Назначение |
|---|---|
| `excel_create_workbook` | новый .xlsx |
| `excel_add_sheet` / `rename_sheet` / `delete_sheet` / `reorder_sheets` | листы |
| `excel_add_column` / `delete_column` / `move_column` | столбцы (formula — опц.) |
| `excel_insert_formula` | формула в ячейку/диапазон |
| `excel_apply_formatting` | стили (цвет фона, шрифт, жирный) + условное форматирование статусов |
| `excel_set_validation` | выпадающий список (Data Validation) в столбец — материализует enum из схемы |
| `excel_read_range` | сырой диапазон — **только отладка** (см. ниже) |
| `excel_validate_formulas` | поиск `#REF!`/`#VALUE!` |

Жёсткие границы:

- **`excel_*` меняет ТОЛЬКО структуру.** Данные в ячейки — никогда здесь, только
  через очередь (Категория 3). Это несущее правило системы.
- **`excel_read_range` — отладочный, НЕ рабочий путь чтения.** Он отдаёт сырой
  2D-массив «посмотреть глазами». Рабочее чтение данных — `json_read_snapshot`
  и проекции (Кат. 3), с экономией токенов и моделью row-by-id. Не тянуть данные
  через `read_range` в обход слоя данных.
- **Защита формул.** `excel_insert_formula` не перезаписывает существующую
  критическую формулу молча — сервер проверяет и отклоняет (как и запись в
  `computed`-столбец на уровне данных).
- **Синхронизация схемы.** После любой ручной правки структуры → перегенерация
  `schema.json` (см. §0, ЗАКОН СИНХРОНИЗАЦИИ).

## 4. КАТЕГОРИЯ 3 — Данные таблиц (JSON State + примитивы)

Механика очереди (как есть, 4 инструмента):

| Инструмент | Назначение |
|---|---|
| `json_read_snapshot` | отдать Claude `read.json` — сжатый контекст без парсинга Excel |
| `json_push_to_queue` | добавить операцию в `pending_updates` write.json |
| `json_execute_queue` | применить очередь к .xlsx → обновить read.json (БЕЗ Qdrant — векторка отложена) |
| `json_clear_queue` | очистить очередь без выполнения (отладка/сброс) |

Пять примитивов (операции над строками — детали в Брифе):

| Примитив | Тип | Назначение |
|---|---|---|
| `table_get_column(table, column)` | чтение | проекция одного столбца `{id: value}` — точечно, экономно |
| `table_get_row(table, row_id)` | чтение | одна строка целиком `{column: value}` |
| `table_set(table, row_id, column, value)` | запись | изменить поле (RMW по строке) |
| `table_append(table, data)` | запись | новая строка, ID присваивает сервер |
| `table_delete(table, row_id)` | запись | удалить строку (с подтверждением) |

Две привязки, без которых слой данных «течёт»:

- **`json_push_to_queue.action` — это ОДИН ИЗ ПЯТИ ПРИМИТИВОВ**, а не произвольная
  свалка действий. Семантические псевдонимы (`update_performance` и т.п.) допустимы
  как сахар, но транслируются в `set`/`append`. Запись идёт только через очередь.
- **Проекции `get_column`/`get_row`** — для точечного ручного чтения данных
  (когда не нужен весь снапшот). Раньше в Категории 3 было только «весь снапшот».

Глубина (row-by-id, read-modify-write, очередь-на-таблицу, защита computed) —
в **Брифе**, здесь не повторяется.

## 5. СТАТУСЫ (enum-столбцы) — сквозная механика

Статус — это **enum-столбец**: фиксированный набор значений со своей палитрой.
Их несколько независимых осей, каждая в своём столбце, со своим набором и цветами.

**Оси статусов (значения + палитра):**

```
HP — системный статус ФАЙЛА (мастер→канал, вариация→видео):
   active   зелёный    файл цел, на месте
   damaged  жёлтый     повреждён
   moved    синий      перемещён
   deleted  серый      soft-delete (обратимо)

Производственный — участие ассета В ВИДЕО (уровень видео):
   under_review  оранжевый   на рассмотрении
   accepted      зелёный     принят в работу
   rejected      красный     отклонён

Статус видео (уровень видео/канал): свой набор (draft/rendering/ready/published…),
своя палитра. Добавляется по той же модели.
```

Цвета по осям не пересекаются по смыслу намеренно: `accepted` зелёный (одобрен)
≠ `active` зелёный (файл жив) — разные столбцы; `rejected` красный (цел, можно
вернуть) ≠ `deleted` серый (помечен к сносу).

**Три куска механики — у каждого свой дом:**

- **Источник истины — `schema.json`** (Бриф). Перечень значений и цвета живут в
  схеме столбца: `{ "type": "enum", "enum": [...], "ui_colors": {...} }`. Только здесь.
- **Витрина в Excel** — генерится ИЗ схемы: `excel_set_validation` вешает
  выпадающий список (нельзя вписать значение вне enum руками), `excel_apply_formatting`
  красит по `ui_colors`. Дропдаун — удобство ручного режима, НЕ источник истины.
- **Смена статуса — это `table_set`, НЕ новый инструмент.** Статус это обычные
  данные; меняется примитивом по ID в столбец статуса, через очередь, как всё.

**`set` со сменой статуса = ПРОВЕРКА, не запуск логики:**

```
table_set(video, META, VID_x, "status", "ready")
   ↓ guard (ops.yaml): допустим ли ПЕРЕХОД? (draft→ready можно, rejected→ready нельзя)
   ↓ запись в очередь → read-modify-write по строке
   ↓ факт: StatusChanged{VID_x, ready} → _SESSION_LOG.md
   ↓ Claude читает факт → выбирает следующий шаг МАРШРУТА
```

- **Guard отвечает «допустимо ли?», НЕ «что теперь делать?».** Привратник, не
  диспетчер. Допустимость перехода — в `ops.yaml`. Маршрут после факта — за Claude.
- **Валидация по enum** не даёт поставить значение вне списка (на уровне сервера,
  не только дропдауна — сервер пишет в обход дропдауна).
- **Отслеживание статусов:** для **чтения/сверки** — да («покажи все `damaged`»);
  для **автозапуска** («стал `ready` → сервер сам запустил монтаж») — НЕТ, это
  жёсткая последовательность, подмена оркестратора. Читать и докладывать можно,
  читать и самому действовать — нельзя.
- **`human_gate`** — только на необратимых переходах (например `purge`: стереть ID
  насовсем). Обратимые (`deleted` = soft-delete) идут обычным `set`. Гейт — guard,
  не инструмент.

### Сигналы и память (частный случай статусов + файл памяти)

**Сигналы** (например «высокая переиспользуемость ассета») — это enum-статус, как
всё выше: скрипт-сверка пишет сигнал в таблицу (`UNIQUENESS_ALERTS`), Claude меняет
его `signal_status` через `table_set`. Гашение ложного сигнала — **смена статуса со
следом** (`dismissed_false_positive`), НЕ стирание. Столбец `flag_source`
(`human_decision` / `ai_check`) фиксирует, чья воля за флагом — защита от тихого
гашения ИИ; `ai_check` на важном сигнале → `human_gate`.

**Память проекта** (`project_memory.md`) — это **файл**, не таблица: пишется
`fs_*`-инструментами, не очередью. Хранит решения и их последствия (замкнутая петля
«сигнал → решение человека → флаг → результат по фактам»). **Не дублирует таблицы** —
только то, чего в них нет (причины и эффекты), ссылки по ID. Два уровня: канал и
видео. Формат — отдельная спека `project_memory`.

## 6. ГРАНИЦА: структура vs данные (главное, что нельзя путать)

| | Категория 2 (`excel_*`) | Категория 3 (`json_*` + примитивы) |
|---|---|---|
| Что меняет | **форму** (листы, столбцы, формулы) | **значения** (строки, ячейки) |
| Как часто | редко (создание/правка схемы) | постоянно (рабочий цикл) |
| Путь записи | напрямую (структура) | только через очередь |
| Лакмус | «меняю скелет таблицы» | «меняю содержимое таблицы» |

Одна фраза: **`excel_*` меняет форму таблицы, очередь меняет её значения.**
Путаница между ними — главный источник ошибок на этом слое.

## 7. РАЗМЕЩЕНИЕ (по ИНСТРУКЦИИ структуры)

Тонкие обёртки → `tools/<category>/`. Движок → `tools/<category>/*_core`.
Реестр операций → `config/ops/<category>.ops.yaml`. Один док на категорию →
`docs/dev/tools/<category>.md` и `docs/github/tools/<category>.md`.
