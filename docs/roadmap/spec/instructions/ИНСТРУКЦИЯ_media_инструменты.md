# ИНСТРУКЦИЯ: Media-инструменты (TTS / STT / картинки)

> Категория `media` в скелете инструментов. Опирается на: ИНСТРУКЦИЯ инструментов
> (скелет, статусы), server_reactions.yaml (три media-сбоя), channel_config
> (RESOURCE_LIMITS), model_routing.yaml (выбор модели/провайдера), ворк-флоу (шаги 6/7/9).
> Термины: книга/лист/столбец. Запись результатов в таблицы — через очередь.

---

## 0. ГЛАВНЫЙ ПРИНЦИП: два типа бэкенда за одной абстракцией

Все media-инструменты — тонкие обёртки поверх `core/providers/`. Но бэкенды **двух
разных природ**, и это определяет всё поведение:

```
core/providers/
  ├─ stt/   stable-ts (ЛОКАЛЬНО)          — Whisper/faster-whisper, на нашей машине
  ├─ tts/   LiteLLM   (ОБЛАКО/локально)   — ElevenLabs/Polly/Gemini/OpenAI…
  └─ img/   LiteLLM   (ОБЛАКО/локально)   — Fal(FLUX/SD)/DALL-E/gpt-image-1…
```

- **STT — локальный** (stable-ts). Платить за прослойку незачем, и он даёт лучшие
  пословные таймкоды + подавление тишины — ровно под сцену→тайминг.
- **TTS/картинки — через LiteLLM** (OpenAI-совместимый шлюз): один интерфейс на много
  провайдеров, конфиг-выбор модели, встроенные fallback'и.
- **LiteLLM спрятан ЗА `core/providers/`** — сменяемый. Завтра другой шлюз → меняем
  адаптер, инструменты и конфиг не трогаем.

## 1. ВЫБОР МОДЕЛИ ПОД КАНАЛ — уже спроектирован, не хардкод

Ничего нового: модель выбирается из **уже существующих** мест.

- `channel_config.RESOURCE_LIMITS` — на канал: `provider` + `fallback_provider` по
  `resource_type` (tts/image/stt).
- `model_routing.yaml` — цепочки fallback и параметры модели. Для STT здесь — **размер
  модели Whisper** (large/medium/base), для TTS/IMG — провайдер + цепочка.

Инструмент НЕ знает модель — он спрашивает её у `core/providers/`, который читает конфиг
канала. Сменить модель на канал = правка конфига, не кода.

## 2. ПАТТЕРН ВЫЗОВА: транспорт и режим (sync/async)

**Транспорт — везде JSON.** Облако (TTS/IMG через LiteLLM): JSON-тело на вход, JSON (или
бинарь/ссылка на файл) на выход, по HTTP. Локально (STT, stable-ts): прямой вызов в
процессе, на выход — JSON-результат. То есть **и вход, и выход — структурированные данные**,
никакого особого транспорта нет.

**Режим — свойство ПРОВАЙДЕРА, не наше решение.** Не всё async. Поэтому инструменты
поддерживают оба режима, а не навязывают очередь задач всем:

```
СИНХРОННЫЙ провайдер (короткий TTS, локальный STT — чаще всего):
   trigger_*  → JSON-запрос → СРАЗУ результат в ответе
   poll_*     → ВЫРОЖДАЕТСЯ (no-op): поллить нечего, ответ уже есть
   download_* → если ответ бинарь — сохранить; verify

АСИНХРОННЫЙ провайдер (картинки FLUX/SD, длинная озвучка):
   trigger_*  → JSON-запрос → {task_id}  (факт TaskStarted)
   poll_*     → сервер САМ опрашивает (TASK_PENDING, Claude не дёргают) → terminal
   download_* → скачать по готовности; verify
```

Async/sync определяет адаптер `core/providers/` по конфигу провайдера. Для sync `poll`
просто отдаёт готовое мгновенно — инструмент тот же, шаг вырождается.

- **Верификация обязательна всегда** (ДОГМА 3, реальность-чек): мало «провайдер сказал
  готово» — проверь файл на диске. Пустой/нулевой файл = сбой, не успех.
- **Именование** при download — по конвенции: `{video_slug}_tts_{scene_id}.wav`,
  `{video_slug}_img_{scene_id}.png`. Путь пишется в таблицу с ID (через очередь).

## 3. СЕМЕЙСТВА ИНСТРУМЕНТОВ

### 3.1. STT — `stt_*` (локально, stable-ts) · ворк-флоу Шаг 7

| Инструмент | Назначение |
|---|---|
| `trigger_transcription` | запустить stable-ts на аудио (модель из конфига канала) |
| `poll_transcription_status` | прогресс локального джоба (`progress_callback`) |
| `parse_timestamps_and_silence` | извлечь пословные таймкоды + границы тишины (VAD) |

- **Вход:** аудио-файл озвучки (`{video_slug}_tts_*.wav`) — локальный вызов, не HTTP.
  **Выход:** JSON с пословными таймкодами и границами тишины → питает лист `SCENES`
  (`timestamp_start/end`) и `TRANSCRIPT`.
- Локальный STT обычно **синхронный** (один вызов), но на длинном аудио идёт с прогрессом
  (`progress_callback`) — тогда `poll` показывает прогресс; на коротком вырождается.
- `suppress_silence=True`, `word_timestamps=True`, `vad=True` — это и есть
  `parse_timestamps_and_silence`, встроено в stable-ts. **Карта тишины — это ВЫХОД STT**
  (что получилось по времени), не вход TTS (см. 3.2).
- **Сбой → `LOCAL_INFERENCE_FAILED`** (не PROVIDER_FAILED): ресурс/среда. Fallback —
  модель поменьше (large→medium→base), не помогло → человек. Контент-отказа тут НЕТ
  (локальная модель не «морализирует»).

### 3.2. TTS — `tts_*` (LiteLLM) · ворк-флоу Шаг 6

| Инструмент | Назначение |
|---|---|
| `prepare_tts_input` | подготовить текст (разбивка, SSML/паузы) из сценария |
| `trigger_tts_generation` | `/audio/speech` через LiteLLM (провайдер из конфига) |
| `poll_tts_status` | статус задачи |
| `download_and_rename_audio` | скачать + имя `{video_slug}_tts_{scene_id}.wav` + verify |

- **Вход:** JSON-запрос (`{model, input, voice, response_format, speed, ...}`) — здесь же
  задаются **паузы/SSML/скорость** (как ОЗВУЧИВАТЬ; это вход TTS, не «карта тишины» из STT).
  `prepare_tts_input` готовит это тело из сценария. **Выход:** аудио (бинарь или ссылка),
  которое потом идёт в STT (3.1) за таймкодами.
- Порядок в ворк-флоу: сначала TTS (6), потом транскрибация озвучки (7).
- Большинство TTS (ElevenLabs, OpenAI tts) **синхронны** — аудио в том же ответе, `poll`
  вырождается. Async — только если провайдер отвечает `task_id` (длинная озвучка).
- **Сбои:** технический → `PROVIDER_FAILED` (сервер: retry+цепочка); контент-отказ →
  `CONTENT_REJECTED` (редко для TTS, но возможен на «запрещённом» тексте).

### 3.3. Картинки — `img_*` (LiteLLM) · ворк-флоу Шаг 9

| Инструмент | Назначение |
|---|---|
| `trigger_image_generation` | `/image/generations` через LiteLLM (модель из конфига) |
| `poll_image_status` | статус задачи |
| `download_and_rename_image` | скачать + имя `{video_slug}_img_{scene_id}.png` + verify |

- **Вход:** JSON-запрос (`{model, prompt, size, n, ...}`) — промпт на ОДИН исходник.
  **Выход:** изображение-исходник (бинарь или ссылка на файл).
- **Что генерится — ИСХОДНИКИ сцены, не сцена целиком.** Диффузия выдаёт строительные
  блоки по отдельности: фон, персонаж, SVG-компоненты. Готовая сцена — это композиция
  исходников (сборка — отдельный этап, вне этого файла).
- **Одна сцена → НЕСКОЛЬКО запросов `img_*`** (по запросу на исходник). Связь
  сцена↔картинка — «один ко многим», не «один к одному». Результаты складываются в
  типизированные списки сцены (`SCENES.bg_assets` / `character_assets` /
  `component_assets`) по `scene_id` — какой тип активен, задаёт `channel_config.SCENE_PROFILE`.
- Дробление на исходники сделано в т.ч. ради **переиспользования**: смена одного
  исходника даёт вариацию сцены без перегенерации остальных (механика — в блоке видео).
- Картинки **чаще async** (FLUX/SD в очереди провайдера): `trigger` → `task_id` → `poll` →
  `download`. Если провайдер синхронный — `poll` вырождается.
- **Сбои — все три природы актуальны, особенно контент-отказ:**
  технический → `PROVIDER_FAILED`; **контент-отказ → `CONTENT_REJECTED`** (модель
  завернула промпт — частый случай для картинок, бывает ложным).

## 4. ТРИ ПРИРОДЫ СБОЯ — у каждой свой «лекарь»

Главное в media — не путать сбои. Три кода, три логики восстановления:

| Сбой | Код | Природа | Кто чинит | Как |
|---|---|---|---|---|
| Локальный инференс | `LOCAL_INFERENCE_FAILED` | ресурс/среда (нет GPU/память/битый файл) | среда | модель↓, потом человек |
| Внешний технический | `PROVIDER_FAILED` | инфраструктура (провайдер лёг/таймаут/лимит) | сервер | retry + авто-цепочка провайдеров |
| Внешний контент-отказ | `CONTENT_REJECTED` | модерация провайдера («не буду») | Claude | переформулировать / сменить вручную / человек |

**Почему контент-отказ отдельно (критично):**
- **Retry бесполезен** — тот же промпт даст тот же отказ. `retry: false`.
- **Смена провайдера — РЕШЕНИЕ Claude, не авто-цепочка** (у провайдеров разная
  модерация; это «поиск, кто согласится», а не «обход поломки»).
- **Часто ложное** — финансовый термин/резкое слово триггерит модерацию зря. Claude
  оценивает: ложное → переформулировать мягче; реально нарушает → человек.
- **Память:** ложный отказ + удачный обход → запись в `project_memory` (что завернули,
  как обошли). Копится знание «провайдер X не любит вот это» — по фактам.

## 5. КАК РЕЗУЛЬТАТ ПОПАДАЕТ В ДАННЫЕ

- Скачанный файл → папка ассетов видео (`assets/audio/`, `assets/...`), имя по конвенции.
- Путь + ID ассета → в таблицу **через очередь** (`table_append`/`table_set`), не напрямую.
- Таймкоды из STT → лист `SCENES` (`timestamp_start/end`) и `TRANSCRIPT` по `scene_id`.
- Сгенерированный ассет получает `system_status: active` (HP) и идёт в обычный жизненный
  цикл (вариации/уникальность/производственный статус — как любой ассет).

## 6. РАЗМЕЩЕНИЕ

Тонкие обёртки → `tools/media/` (`stt_*`, `tts_*`, `img_*`). Адаптеры бэкендов →
`core/providers/{stt,tts,img}/`. Реестр операций → `config/ops/media.ops.yaml`.
Выбор модели → `channel_config.RESOURCE_LIMITS` + `model_routing.yaml`. Реакции —
`server_reactions.yaml` (PROVIDER_FAILED / CONTENT_REJECTED / LOCAL_INFERENCE_FAILED).

## 7. ЧТО ПРОВЕРИТЬ ПЕРЕД БОЕМ (честные оговорки)

- **Покрытие провайдеров LiteLLM** широкое, но не бесконечно — проверь, что ИМЕННО твой
  TTS/IMG-провайдер в списке (ElevenLabs/Deepgram/Polly/Gemini/Fal/DALL-E — есть).
- **stable-ts** требует FFmpeg + PyTorch (GPU желателен). Репозиторий архивирован
  (разработка на паузе), но рабочий и стабильный; faster-whisper/MLX поддержаны.
- **Контент-отказ** у разных провайдеров срабатывает по-разному — цепочка fallback для
  картинок может реально помочь (другой провайдер мягче), но это решение Claude.
