# ИНСТРУКЦИЯ: Шаблоны (рабочее пространство + таблицы)

> Закрывающий документ блока. Опирается на: ИНСТРУКЦИЯ структуры (где что лежит),
> ИНСТРУКЦИЯ инструментов (fs_* / excel_* / примитивы / статусы), Бриф (schema.yaml),
> server_reactions.yaml (STRUCTURE_INCOMPLETE и пр.). Термины: книга/лист/столбец/строка.

---

## 0. ЧТО ЗАДАЁТ ШАБЛОН, А ЧТО ГЕНЕРИТ ИНТРОСПЕКТОР

Важное разделение, чтобы не уйти в недели ручной работы:

- **Формат шаблона + порядок создания + новые листы/столбцы** — задаём здесь, руками.
- **Столбцы ~90 существующих листов** — НЕ переписываем вручную. Их вытаскивает
  `scripts/introspect_tables.py` из готовых Excel-книг → генерит `schema.yaml` каждой.
- **Руками спекаем ТОЛЬКО то, чего в Excel ещё нет:** лист `SCENES`, статус-столбцы
  (HP и производственный). Интроспектор их не найдёт — их там пока нет (см. АУДИТ).

Итог: шаблон таблицы = «список листов книги» + ссылка на их схемы (генерятся), плюс
явно дописанные новые листы/столбцы. Не свалка из 90 листов.

## 1. ФОРМАТ ШАБЛОНА (общий для папок, файлов, листов)

Каждый фрагмент несёт свой `required` ВНУТРИ себя (не отдельным списком):

```yaml
# template_video.yaml — пример фрагмента
video:
  folders:
    - { name: "assets/svg",         required: true }
    - { name: "assets/scenes",      required: true }
    - { name: "assets/audio",       required: true }
    - { name: "assets/transitions", required: true }
  files:
    - { name: "video_data.xlsx", kind: table, table_template: video_data, required: true }
    - { name: "read.json",       kind: file,  required: true }
    - { name: "write.json",      kind: file,  required: true }
  children: []        # видео — лист дерева, детей-сущностей нет
```

Правила формата:
- **`required` внутри фрагмента** — источник истины об обязательности один, в самом фрагменте.
  валидация сверяет созданное с фрагментами, где `required: true`.
- **ID присваивает сервер** каждому фрагменту при создании (имя — на входе, ID — в `facts`).
- **Композиция по ссылке.** Родитель знает ИМЯ типа ребёнка, не его устройство:
  `channel.children: [{ type: video, container: "videos/" }]`. Ребёнок про родителя
  не знает — родитель приходит как `parent_ids` снаружи при создании.
- **Пофрагментный скип.** Нет валидного имени → фрагмент пропускается, соседи создаются,
  Claude получает факт о пропуске (код `STRUCTURE_INCOMPLETE`). Не атомарный откат.

## 2. ТРЁХФАЗНОЕ СОЗДАНИЕ (с двойной проверкой)

Создание канала (и любой сложной сущности) идёт **тремя фазами**, не одним вызовом.
Порядок вынужден зависимостью данных: **в таблицу пути пишутся уже с ID, значит
папки (с их ID) должны существовать раньше таблиц.**

```
ФАЗА 1 — ПАПКИ
   материализовать все folder-фрагменты (по композиции) → каждая получает ID
   ПРОВЕРКА ДВУМЯ СПОСОБАМИ:
     (1) опрос инструмента: facts FolderCreated с ID (цикл/поллинг)
     (2) физическое подтверждение: fs_get_directory_tree / fs_smart_search читает ДИСК
   ГЕЙТ: не идём в фазу 2, пока папки не подтверждены ОБА раза
        ↓
ФАЗА 2 — ТАБЛИЦЫ
   теперь папки+ID есть → создаём книги (.xlsx), пишем пути с ID в нужные листы
   excel_create_workbook → add_sheet/add_column по схеме → set_validation+formatting (статусы)
   ПРОВЕРКА: те же два способа + excel_validate_formulas (нет #REF!)
        ↓
ФАЗА 3 — ФАЙЛЫ / СКРИПТЫ
   листовые артефакты: read.json, write.json, project_memory.md, _INDEX.md и пр.
   могут ссылаться на ID, уже записанные в таблицы
   ПРОВЕРКА: те же два способа
```

- **Claude оркеструет фазы, сервер валидирует каждую.** Сервер не гонит
  последовательность сам — Claude делает фазу, читает результат, делает следующую.
- **Двойная проверка = реальность-чек.** Слово инструмента (facts) +
  независимое чтение диска. Расхождение → не идём дальше, сигналим.
- **На любой фазе пофрагментный скип в силе:** нет имени сетки → сетка пропущена,
  но канал создаётся (его parent — не сетка напрямую, родительские ID опциональны по уровню).

## 3. ШАБЛОНЫ РАБОЧЕГО ПРОСТРАНСТВА — НАШИ

Иерархия (композиция по ссылке, каждый знает только имя типа ребёнка):

```
niche  → networks → channel → video
```

Дерево наших уровней (что физически создаётся):

```
niches/
  {ниша}/
    _NICHE_INDEX.md
    niche_read.json
    niche_write.json
    networks/
      {сетка}/
        _NETWORK_INDEX.md
        network_dashboard.xlsx        (table)
        network_config.xlsx           (table)
        channels/
          {наш_канал}/                ← channel
            channel_data.xlsx         (table)
            channel_config.xlsx       (table)
            project_memory.md
            scene_layouts/
            videos/
              {видео}/                ← video
                video_data.xlsx       (table)
                read.json
                write.json
                project_memory.md
                assets/{svg,scenes,audio,transitions}
                renders/
        competitors/                  ← вторая ветка сетки (см. §4)
```

| Шаблон | folders / files (required) | children (ссылка) |
|---|---|---|
| `niche.tpl` | `_NICHE_INDEX.md`, `niche_read.json`, `niche_write.json` | `{type: network, container: networks/}` |
| `network.tpl` | `_NETWORK_INDEX.md`, `network_dashboard.xlsx`(table), `network_config.xlsx`(table) | `{type: channel, container: channels/}` + `{type: competitor_channel, container: competitors/}` |
| `channel.tpl` | `channel_data.xlsx`(table), `channel_config.xlsx`(table), `project_memory.md`, `scene_layouts/` | `{type: video, container: videos/}` |
| `video.tpl` | `video_data.xlsx`(table), `read.json`, `write.json`, `project_memory.md`, `assets/{svg,scenes,audio,transitions}`, `renders/` | — (лист дерева) |

Примечание: `network` имеет ДВЕ ветки детей — наши `channel` и `competitor_channel`
(см. §4). Это и есть точка, где конкуренты подвешиваются к сетке через наш канал.

## 4. ШАБЛОНЫ КОНКУРЕНТОВ

Особенность: конкуренты **группируются по нашему каналу** — лишний уровень вложенности.

```
competitors/
  {наш_канал}/                       ← контейнер: для какого нашего канала анализ
    {канал_конкурента}/              ← competitor_channel
      channel_meta.json
      competitor_channel_data.xlsx   (table)
      competitor_channel_read.json
      competitor_channel_write.json
      videos/
        {видео_конкурента}/          ← competitor_video
          competitor_video_data.xlsx (table)
          read.json
          write.json
```

| Шаблон | files (required) | children |
|---|---|---|
| `competitor_channel.tpl` | `channel_meta.json`, `competitor_channel_data.xlsx`(table), `competitor_channel_read.json`, `competitor_channel_write.json` | `{type: competitor_video, container: videos/}` |
| `competitor_video.tpl` | `competitor_video_data.xlsx`(table), `read.json`, `write.json` | — |

- Родитель `competitor_channel` — это **наш канал** (контейнер `competitors/{наш_канал}/`),
  а не сетка напрямую. В `network.tpl` это вторая ветка `children`.
- Имена файлов с префиксом `competitor_`, парные `competitor_channel_read/write.json`.
- `niche_network_data.xlsx` (агрегат по конкурентам ниши) живёт на уровне `competitors/`.

## 5. ШАБЛОНЫ ТАБЛИЦ (книга → листы → столбцы)

### 5.1. Существующие книги — через интроспектор

Формат шаблона книги — список листов; столбцы каждого листа берутся из `schema.yaml`,
сгенерированной интроспектором из реального Excel. Книги:

| Книга | Уровень | Листов | Источник столбцов |
|---|---|---|---|
| `video_data.xlsx` | видео (наш) | ~22 | интроспектор + новые (5.2, 5.3) |
| `channel_data.xlsx` | канал (наш) | ~15 | интроспектор + статус мастеров (5.3) |
| `network_dashboard.xlsx` | сетка | — | интроспектор |
| `competitor_video_data.xlsx` | видео конкурента | ~19 | интроспектор |
| `competitor_channel_data.xlsx` | канал конкурента | ~10 | интроспектор |
| `niche_network_data.xlsx` | ниша/конкуренты | ~12 | интроспектор |

> Интроспектор: помечает столбцы с `=` как `computed:true, writable:false`; остальные
> `writable:true`. Человек после генерации проверяет computed-флаги. НЕ переписываем руками.

### 5.2. НОВЫЙ ЛИСТ `SCENES` (книга video_data) — спекаем руками

Отдельный лист (не секция UNIQUENESS). Точка сборки сцены: тайминг + состав + привязка.

| Столбец | Тип | Прим. |
|---|---|---|
| `scene_id` | string (id) | ключ сцены |
| `act_id` | string (fk) | привязка к акту/главе сценария |
| `phrase_ref` | string (fk) | ссылка на фразу в `TRANSCRIPT` (точная привязка) |
| `start_time` | float | из таймкодов `TRANSCRIPT` (после озвучки) |
| `end_time` | float | из таймкодов `TRANSCRIPT` — даёт ограничение сцены по тайму |
| `asset_ids` | string[] | состав: svg + audio + transition (не только svg) |
| `scene_uniqueness` | float (computed) | уникальность комбинации (svg+аудио+переходы) |

В листе `UNIQUENESS` Секцию-3 (scene) убрать, оставить ссылку на итог `SCENES`.
`scene_id` — join-ключ к `TRANSCRIPT` / `ASSETS_USED` / `SCENE_SEMANTICS`.

### 5.3. НОВЫЕ СТАТУС-СТОЛБЦЫ — спекаем руками (enum + палитра в схеме)

**HP (системный статус файла)** — двухуровнево по владению:
- мастера → лист статусов на **канале** (рядом с `ASSET_USAGE_HISTORY`);
- вариации → лист статусов на **видео** (рядом с `ASSETS_USED`).

```
enum: active(зелёный) / damaged(жёлтый) / moved(синий) / deleted(серый, soft-delete)
```

**Производственный статус ассета** — на **видео** (в `ASSETS_USED`):

```
enum: under_review(оранжевый) / accepted(зелёный) / rejected(красный)
```

**Статус видео** — на видео (META) и сводно на канале (VIDEOS_INDEX):

```
enum: draft / in_production / rendering / ready / published / archived  (палитра своя)
```

Все статусы: `enum` + `ui_colors` живут в `schema.yaml` (источник истины);
в книге материализуются `excel_set_validation` (дропдаун) + `excel_apply_formatting`
(цвета). Смена — `table_set` по ID. Подробности механики — ИНСТРУКЦИЯ инструментов §5.

## 6. РАЗМЕЩЕНИЕ

Шаблоны рабочего пространства → `config/templates/workspace/*.tpl.yaml`
(niche, network, channel, video, competitor_channel, competitor_video).
Схемы таблиц → `config/templates/tables/*.schema.yaml` (генерит интроспектор +
ручная правка для `SCENES` и статусов). Движок шаблонов → `core/engine/`,
тонкие обёртки создания → `tools/filesystem/` и `tools/excel_engine/`.
