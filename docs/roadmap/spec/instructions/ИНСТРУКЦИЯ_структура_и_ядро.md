# ИНСТРУКЦИЯ: Структура проекта и ядро сервера

> Это **карта и закон**, а не пересказ. Детали инструментов, реакций и таблиц
> живут в своих файлах (см. раздел «Карта артефактов»). Здесь — где что лежит,
> почему именно там, и что с чем нельзя смешивать.

---

## 0. ГЛАВНЫЙ ЗАКОН: у каждого файла ровно один дом

Структура папок — это **контракт размещения**, а не оформление. Правило одно:

> Прежде чем создать файл, ответь: «что это за сущность?».
> У ответа есть **ровно один** легальный каталог. Другого места нет.

Из этого следует жёсткое разделение, которое **нельзя нарушать**:

| Тип сущности | Где живёт | Где НЕ живёт |
|---|---|---|
| **Документация** | только `docs/` | никогда рядом с кодом |
| **Декларации** (поведение: yaml/schema) | только `config/` | никогда в `.py` |
| **Бизнес-логика** (ядро/инструменты/пайплайн) | `core/`, `tools/`, `pipeline/` | не в `scripts/`, не в `config/` |
| **Утилиты обслуживания** (интроспектор схем) | только `scripts/` | не в `pipeline/`, не в `core/` |
| **Тесты** | только `tests/` | никогда рядом с кодом |
| **Ассеты** (svg, audio, scenes — продукт работы) | в данных, в `assets/` | **никаких `src/`** — это данные, а не исходники |
| **Техкаталоги** (`src`, `.venv`, `__pycache__`, `dist`) | только в `mcp_server/` (и в `.gitignore`) | **никогда в данных** — их наличие там = перепутаны вселенные |

Четыре вещи, которые путают чаще всего, и которые здесь разведены навсегда:
**документация ≠ код**, **поведение (config) ≠ код**, **утилита ≠ бизнес-логика**,
**сервер ≠ данные, которыми он управляет** (см. §1).

## 1. ДВЕ ВСЕЛЕННЫЕ: сервер и рабочее пространство

Не смешивать. Это разные деревья.

```
mcp_server/        ← ЭТОТ репозиторий. Код, конфиги, доки. Версионируется в git.
workspace/         ← ДАННЫЕ, которыми сервер управляет. НЕ в git сервера.
└─ niches/
   └─ <niche>/                       _NICHE_INDEX.md, niche_read.json, niche_write.json
      └─ networks/
         └─ <network>/               _NETWORK_INDEX.md, network_dashboard.xlsx, network_config.xlsx
            ├─ channels/             ← НАШИ каналы
            │  └─ <channel>/         channel_data.xlsx, channel_config.xlsx,
            │     │                  project_memory.md, scene_layouts/
            │     └─ videos/
            │        └─ <video>/     video_data.xlsx, read.json, write.json,
            │                        project_memory.md, assets/{svg,scenes,audio,transitions}
            └─ competitors/          ← КОНКУРЕНТЫ. niche_network_data.xlsx (агрегат по нише)
               └─ <наш_канал>/       (конкуренты сгруппированы по нашему каналу)
                  └─ <конкурент>/    competitor_channel_data.xlsx, channel_meta.json,
                     │               competitor_channel_read.json, competitor_channel_write.json
                     └─ videos/
                        └─ <видео>/  competitor_video_data.xlsx, read.json, write.json
```

Полная иерархия сверху вниз: **ниша → сетка → (наши каналы | конкуренты) → видео → ассеты**.
Детальные шаблоны каждого уровня — в ИНСТРУКЦИИ шаблонов; здесь — каноничная карта данных.

Путь к `workspace/` сервер читает из конфига (относительный путь внутри проекта),
а не хардкодит. Сервер можно переустановить, не тронув данные; данные можно
перенести, не трогая сервер.

**Память проекта — данные, не код.** `project_memory.md` живёт в `workspace/` на
двух уровнях по иерархии (рядом с каналом и рядом с видео), как `read.json`. Ведёт
её state-слой (`core/state/`), но физически она лежит в данных. Память хранит
**решения и их последствия**, НЕ дублирует таблицы (ссылки по ID). Формат — см.
отдельную спеку `project_memory`.

**Раскладки сцен — тоже данные канала.** `scene_layouts/` (слоты: где стоят фрагменты,
их габариты) живёт в `workspace/` на уровне канала, рядом с `channel_config`. Это
родственник `SCENE_PROFILE` (тот — *какие* типы активны, раскладка — *где* стоят).
Обёртки FFmpeg, которые это применяют, — наоборот, КОД (`core/providers/ffmpeg/`).
Формат и движок — см. ИНСТРУКЦИЯ видеомонтажа.

**Появление техкаталога в данных = перепутаны вселенные.** Если в `workspace/`
обнаружен `src/`, `.venv/`, `__pycache__/`, `dist/`, `node_modules/` — это не
«неудачное место», а сигнал, что ИИ принял данные за исходники кода. Ловит
фоновая сверка (фоновый процесс сервера) → код `INTRUSIVE_DIR` в реестре
реакций. Ассеты видео — это **данные**, им `src/` не нужен и запрещён.

**Изоляция от системы (жёстко).** Проект самодостаточен: все зависимости живут
внутри него (виртуальное окружение внутри `mcp_server/`, в `.gitignore`).
Сервер **не трогает системные пакеты, не требует повышенных прав, не пишет за
пределы своих двух деревьев.** Никаких системных путей и привилегий — ни в коде,
ни в путях, ни в инструкциях. Всё, что нужно серверу, лежит у него внутри.

> **ЗАКОН РАЗМЕЩЕНИЯ ДОКОВ (чтобы не теряться).** Два правила:
> 1. **Доки зеркалят код.** `docs/dev/tools/<X>` лежит параллельно `tools/<X>` или
>    `core/providers/<X>`. Документ ищется там, где искался бы сам код — не нужно гадать.
> 2. **Масштаб по сложности.** Простой инструмент/категория → ОДИН `.md` (`filesystem.md`).
>    Сложная способность (адаптер с частями, несколько обёрток, провайдер) → ПАПКА:
>    `overview.md` (общая картина) + по файлу на часть (`stt.md`, `ffmpeg_adapter.md`,
>    `wrappers/<name>.md`). Так у крупного есть место дышать, мелкое не дробится зря, а
>    быстрый доступ к нужному куску сохраняется.

## 3. ПОЧЕМУ ИМЕННО ТАК (3 решения, на которые опирается структура)

**Декларации отделены от кода (`config/` ≠ `core/`).** Поведение — операции,
реакции, шаблоны, маршрутизация — это данные. Код их только исполняет. Сменить
модель, реакцию, структуру сущности = правка yaml, не Python. Никакого хардкода.

**Один движок под категорией, тонкие обёртки сверху (`core/engine` + `tools/*`).**
Сотни инструментов — это счётчик интерфейса, а не объём кода. Логика категории
в одном `*_core`, наружу торчат тонкие контракты. Кода мало, контракты строгие.

**Точки входа — наборы предусловий, шаги — единая библиотека (`pipeline/`).**
4 входа не дублируют воркфлоу: они проверяют, какие данные уже есть, и роняют
Claude в нужную точку общей библиотеки шагов. Производственные шаги (сценарий →
TTS → … → рендер) описаны ОДИН раз и переиспользуются всеми входами.

## 4. ЯДРО + ТУННЕЛЬ (кто кем управляет)

```
Claude (веб) ──туннель──▶ server.py (MCP) ──▶ core/engine ──▶ tools/* ──▶ workspace/
   ▲                                                │
   └──────────── ToolResult / ErrorDetail ──────────┘
```

- **Claude — оркестратор.** Думает, выбирает следующий шаг, ведёт диалог с человеком.
- **Сервер — исполнитель и валидатор.** Не диктует порядок, не принимает смысловых
  решений. Проверяет предусловия (наличие данных), исполняет, возвращает факт.
- **Граница — MCP-контракты.** Claude видит тонкие инструменты с Pydantic-схемами.
  Всё, что сложно (guards, реакции, очереди, ID), скрыто за этой границей.
- **Туннель** публикует `server.py` наружу, чтобы Claude-веб мог дёргать инструменты.

Поток одного вызова: `Claude → контракт инструмента → guards (ops.yaml) → handler →
ToolResult{facts}`. При сбое: `code → server_reactions.yaml → ErrorDetail{recovery}` →
Claude читает recovery и делает следующий ход. Состояние процесса — в `core/state`,
НЕ внутри инструментов (инструмент чистый, без памяти).

## 5. КАРТА АРТЕФАКТОВ (что написано → что ИИ создаёт и куда)

> Это **контракт создания**: по нему ИИ-разработчик разворачивает каталоги, скрипты и
> конфиги. Если артефакта тут нет — ИИ его не создаст. Покрыто всё, что спроектировано.

### 5.1. Данные, слой таблиц, состояние

| Артефакт | Что это | Что СОЗДАТЬ и где |
|---|---|---|
| **Бриф табличных инструментов** | спека слоя данных (5 примитивов, row-by-id, очередь) | код → `tools/tables/`; спека → `docs/dev/tools/tables.md`; схемы → `config/templates/tables/` |
| **server_reactions.yaml** | реестр реакций (5 классов + DEFAULT, 13 кодов) | `config/server_reactions.yaml`; читалка → `core/reactions/` |
| **Таблицы и джейсон** (исходник) | описание 92 листов и формул | источник для `scripts/introspect_tables.py` → генерит `config/templates/tables/*.schema.yaml` |
| **8 схем книг** (`*.schema.md`) | финальные схемы книг (листы/столбцы/флаги) | человекочитаемо → `docs/dev/tables/`; машинно → `config/templates/tables/<book>.schema.yaml` |
| **project_memory.spec** | формат памяти (решения+последствия) | state-слой `core/state/` ВЕДЁТ; файлы `project_memory.md` → в `workspace/` на уровне канала И видео |

### 5.2. Инструменты и структура

| Артефакт | Что это | Что СОЗДАТЬ и где |
|---|---|---|
| **ИНСТРУКЦИЯ инструменты** | скелет + категории fs/excel/tables + статусы | `tools/filesystem/`, `tools/tables/`, `tools/excel_engine/`; операции → `config/ops/<cat>.ops.yaml` |
| **ИНСТРУКЦИЯ шаблоны** | формат шаблонов + 3-фазное создание | `config/templates/workspace/*.tpl.yaml` (niche/network/channel/video/competitor_channel/competitor_video); движок → `core/engine/` |
| **Туннель к Claude** | как ИИ-оркестратор подключается к серверу | MCP-эндпоинт → `server.py` (точка входа); установка/keepalive туннеля → `core/transport/` |

### 5.3. Media — генерация (озвучка / транскрибация / картинки)

> Один закон: **способность = адаптер за `core/providers/` + тонкие инструменты `tools/media/`
> + операции `config/ops/media.ops.yaml`.** Бэкенды разной природы, но интерфейс наружу один.

| Способность | Адаптер (КОД) | Инструменты (КОД) | Бэкенд |
|---|---|---|---|
| Транскрибация (STT) | `core/providers/stt/` | `tools/media/stt_*` | stable-ts, ЛОКАЛЬНО |
| Озвучка (TTS) | `core/providers/tts/` | `tools/media/tts_*` | LiteLLM (облако/локально) |
| Картинки (IMG) | `core/providers/img/` | `tools/media/img_*` | LiteLLM (облако/локально) |

Выбор модели на канал → `channel_config.RESOURCE_LIMITS` + `model_routing.yaml`. Сбои →
`PROVIDER_FAILED` / `CONTENT_REJECTED` / `LOCAL_INFERENCE_FAILED`. Спека → `ИНСТРУКЦИЯ media`.

### 5.4. Видеомонтаж — сборка сцены (НЕСКОЛЬКО уровней, не путать код и данные)

> Самое дробное место — здесь у видео ЧЕТЫРЕ дома по двум вселенным. Если не перечислить
> явно, видео «окажется где попало». Поэтому разложено построчно:

| Часть | Вселенная | Что СОЗДАТЬ и где |
|---|---|---|
| Обёртки FFmpeg + адаптер | КОД (git) | `core/providers/ffmpeg/wrappers/` + `core/providers/ffmpeg/ffmpeg_adapter.py` |
| Инструменты монтажа | КОД (git) | `tools/video/` (набор — после подбора обёрток) |
| Раскладки сцен (слоты/позиции) | ДАННЫЕ канала | `workspace/.../channel/scene_layouts/` (родственник `SCENE_PROFILE`) |
| Параметры рендера (codec/res/fps/crf) | ДАННЫЕ канала | лист `RENDER_CONFIG` в `channel_config.xlsx` |

Возможности кодеков (H.265/ProRes) — встроены в FFmpeg, **отдельного каталога пресетов НЕТ**.
Сбой монтажа — локальный (не PROVIDER_FAILED). Спека → `ИНСТРУКЦИЯ видеомонтажа`.

### 5.5. Прочее

| Артефакт | Дом |
|---|---|

## 6. README И ПРОЦЕСС УСТАНОВКИ/ЗАПУСКА (требования)

Это требования к оформлению, НЕ дерево. Дерево лишь показывает, что файлы есть;
как они устроены — здесь.

**README.md (в корне) содержит три части, в этом порядке:**
1. **Структура проекта** — краткая карта каталогов (что где живёт: `core/`, `tools/`,
   `config/`, `workspace/`), чтобы новый человек за минуту понял расклад.
2. **Установка** — блок команд (см. ниже), одним куском, копируется и выполняется.
3. **Запуск** — отдельная команда запуска (см. ниже).

**Принцип: одна команда из корня.** И установка, и запуск выполняются из корня проекта
одной командой каждая. Никаких «зайди туда, активируй это, потом набери то» вручную —
всё внутри `install.sh` / `run.sh`.

**`install.sh` — установка + первичный запуск (выполняется ОДИН раз):**
- создать локальный `.venv` внутри проекта (систему не трогаем);
- поставить зависимости из `requirements.txt`;
- первичная инициализация (что нужно перед первым стартом);
- по завершении — готовность к запуску.
Требование: идемпотентность — повторный запуск не ломает уже установленное.

**`run.sh` — запуск (когда уже установлено):**
- активировать локальный `.venv`;
- стартовать `server.py` (MCP-эндпоинт + туннель).
Требование: не делает установку и не лезет в систему — только поднимает сервер.

**Граница между ними:** `install.sh` готовит окружение (редко, один раз);
`run.sh` поднимает сервер (часто). Разделение — чтобы обычный старт был мгновенным
и не трогал зависимости.

## 7. ПРАВИЛО ДЛЯ ИИ-РАЗРАБОТЧИКА (одно, короткое)

Перед созданием любого файла — определи его тип по таблице из §0 и положи в
единственный легальный каталог. Если кажется, что файл «и логика, и утилита» —
он не туда спроектирован: раздели на два, каждый в свой дом. Документацию не
клади рядом с кодом. Поведение не зашивай в Python — выноси в `config/`.
Состояние не храни в инструментах — только в `core/state/`.
