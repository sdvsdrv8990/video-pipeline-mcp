# ИНСТРУКЦИЯ: Видеомонтаж (сборка сцены)

> Сборка готовой сцены из исходников. Опирается на: ИНСТРУКЦИЯ media (исходники приходят
> оттуда), video_data.SCENES (типизированный состав сцены), channel_config (SCENE_PROFILE +
> рендер), ИНСТРУКЦИЯ структуры (две вселенные), server_reactions. Термины: книга/лист/столбец.
>
> ⚠️ Это инструкция-основа блока. Конкретные FFmpeg-обёртки/команды выбираются ОТДЕЛЬНО
> (следующий шаг — подбор реп под наши операции). Здесь — архитектура, не реализация команд.

---

## 0. РОЛЬ И ГРАНИЦА

Монтаж **компонует** сцену из готовых исходников — он их не генерит.

```
media (img_*/tts_*)  →  ИСХОДНИКИ (фон, персонаж, компоненты, музыка, звук)
   ↓
МОНТАЖ (этот файл)   →  компоновка: слои + позиция + цветокор/фильтры + аудио + переходы
   ↓
готовая сцена → видео
```

- Диффузия даёт **строительные блоки**, монтаж **собирает** из них кадр. Разные этапы.
- **Claude НЕ подключён к FFmpeg напрямую.** Управление — через скрипты и пресеты-раскладки:
  сцена превращается в применяемую раскладку, Claude управляет назначением фрагментов и
  параметрами, а не дёргает FFmpeg в реальном времени.

---

## 1. ДВИЖОК: FFmpeg за `core/providers/`

**Выбран FFmpeg, не DaVinci.** Причина: FFmpeg headless, скриптуемый, детерминированный —
ровно «сцена → пресет → применяй автоматически». DaVinci/resolve-mcp — это живой контроль
GUI-приложения в реальном времени, против пресетной автоматизации и не запускается на сервере
headless. FFmpeg создавался как основа видеопродакшена и умеет нужное из коробки.

```
core/providers/ffmpeg/          # КОД (git)
   ├─ wrappers/                 # под-каталоги обёрток/реп (несколько — норма)
   │   ├─ wrapper_a/
   │   └─ wrapper_b/
   └─ ffmpeg_adapter.py         # сводит обёртки в ЕДИНЫЙ интерфейс наружу
```

- **Несколько обёрток на бэкенде — норма; несколько обёрток в лице ИИ — хаос.** Claude видит
  один набор media/монтаж-инструментов, чем исполнено внутри — деталь адаптера.
- **Выбор обёртки — по надёжности и тестам, не по числу команд.** Свой тонкий адаптер под
  свои операции (оверлей/фильтр/аудио/рендер) — допустимый и часто лучший путь; FFmpeg делает
  работу, мы формируем команды. Писать движок с нуля — НЕ нужно.

---

## 2. ПОЗИЦИОНИРОВАНИЕ: слоты поверх пикселей

**Узкое место всего блока:** у ИИ нет пространственного восприятия — он не «видит» холст и
угадывает координаты. Движок тут ни при чём (он исполнитель), проблема в том, кто считает «куда».

Решение — **два уровня, не два режима:**

```
БАЗА:    всё в пикселях (FFmpeg работает в них; слот = пиксельный прямоугольник + габариты)
СВЕРХУ:  сетка/слоты — удобная РУКОЯТКА над пикселями (центр / низ / справа / правая половина)
ДОСТУП:  ИИ обычно мыслит СЛОТАМИ (безопасно, без угадывания);
         ОПУСКАЕТСЯ в пиксели, когда нужно растянуть/сдвинуть точно
```

Не «или сетка, или пиксели», а **слот = именованный пиксельный прямоугольник, который можно
переопределить вручную**. Один источник (пиксели), два уровня доступа — ИИ не мечется. Фрагмент
масштабируется под габариты слота, а не ИИ угадывает размер.

---

## 3. РАСТЯГИВАНИЕ = ВАРИАЦИЯ

Пиксельная правка размера/позиции фрагмента — это для редактора секунда, а для системы —
**новая вариация** того же мастера. Замыкается на уже построенное:

- растянул/сдвинул фрагмент → новый `variation_id` того же `global_asset_id`;
- попадает в `video_data.ASSETS_USED` (HP вариации, производственный статус) и в
  `scene_uniqueness` (взвешенная композиция);
- вариация без новой генерации — дёшево, и система это уже учитывает мягким весом.

---

## 4. ДВА РЕЖИМА: создание vs применение

Самое дорогое — **первичное создание раскладки** (ты это верно предсказал). Развести:

```
СОЗДАНИЕ РАСКЛАДКИ (дорого, редко):  слотов нет → задать раскладку с нуля
   → максимум усилий: человек + ИИ + проверка, возможна ручная правка
   → результат: пресет-раскладка (слоты: координаты + габариты)

ПРИМЕНЕНИЕ РАСКЛАДКИ (дёшево, часто): слоты есть → ИИ назначает фрагменты в слоты
   → почти бесплатно: назначение, не вычисление координат
```

Вся пространственная боль концентрируется в создании и **платится один раз на тип сцены** →
сотни сцен наследуют слоты. Та же экономия, что с переиспользованием исходников, но на уровне
раскладки: один пресет → много сцен. «Физический размер фрагмента» решается тут же — слот задаёт
макс. габариты, фрагмент масштабируется под слот.

## 5. ГДЕ ЧТО ЛЕЖИТ (две вселенные — строго)

| Артефакт | Где | Природа |
|---|---|---|
| Обёртки FFmpeg + адаптер | `core/providers/ffmpeg/` | КОД (git) |
| Раскладки сцен (слоты, позиции) | `workspace/.../channel/scene_layouts/` | ДАННЫЕ канала (не git) |
| Параметры рендера (codec/res/aspect/fps/crf) | `channel_config` (лист `RENDER_CONFIG`) | данные канала |
| Возможности кодеков (H.265, ProRes…) | внутри FFmpeg | НЕ отдельный каталог |

- **Раскладки сцен — ДАННЫЕ, не код.** Это родственник `SCENE_PROFILE`: профиль говорит
  *какие* типы фрагментов активны, раскладка — *где* они стоят. Обе — на уровне канала.
- **Параметров рендера — на канал** (решение: каждый канал рендерит по-своему; вертикальный
  short ≠ горизонтальный long). Лист `RENDER_CONFIG` в `channel_config`. Если позже понадобится
  общий дефолт — он ляжет в `network_config` с переопределением на канале (задел есть, не сейчас).
- **Отдельного каталога пресетов кодеков НЕТ.** H.265/4K/CRF — встроены в FFmpeg, задаются
  строками параметров в `RENDER_CONFIG`, не хранилищем.

## 6. ПЕРЕИСПОЛЬЗОВАНИЕ = ЭКОНОМИЯ

Сцена это композиция исходников по ссылкам (`SCENES.bg_assets`/`character_assets`/…). Смена
**одного** фрагмента (другой персонаж на том же фоне) даёт другую сцену без перегенерации
остальных → меньше запросов к диффузии, меньше токенов. Композиция и экономия — одно свойство.

---

## 7. ВЫХОД РЕНДЕРА В ПРОЕКТ (имя, место, привязка)

Смонтированное видео обязано **финишировать в проекте** — с именем от `video_id`, в своём
доме, связанным с данными. Иначе рендер анонимен (`ORPHAN`) и выпадает из иерархии.

**Стадии (не один файл):** черновой → финальный (плюс, возможно, форматы). Каждый рендер —
строка в листе `video_data.RENDERS` (`render_stage` draft/final, профиль из `RENDER_CONFIG`,
путь, статус, привязка финала к черновику).

**Место:** `videos/<video>/renders/` — отдельно от `assets/`-сырья. Готовый продукт не
мешается с исходниками.

**Имя:** выводится из проекта, не из движка: `{video_slug}_{render_stage}[_{format}].mp4`.
`video_slug`/`video_id` — из `video_data.META`.

**Как файл попадает на место — один принцип, БЕЗ развилки онлайн/локал:**

```
УРОВЕНЬ 1 (предпочтительно): инструмент принимает output_path
   → пишем сразу в videos/<video>/renders/ с нужным именем. Готово.
УРОВЕНЬ 2 (фолбэк, если путь задать нельзя): сервер ЗНАЕТ, куда инструмент кладёт
   → перемещает в проект + переименовывает. Онлайн/локал не различаем.
```

Различие онлайн/локал тут не нужно: как только файл существует, важно лишь «где он лёг».
FFmpeg — лучший случай (своя обёртка → `output_path` задаём сами, всегда уровень 1).
**Умеет ли инструмент `output_path`** — свойство адаптера, выясняется при подборе реп.

**Verify обязателен:** для видео мало «файл есть» — нужно «файл валидный» (≠ 0 байт,
играбелен). Результат → `RENDERS.file_verified`. Путь пишется в таблицу через очередь.

## 8. СБОИ (стыковка с реакциями)

FFmpeg — **локальный** движок, внешнего провайдера нет. Значит сбой монтажа — **не**
`PROVIDER_FAILED` (там цепочка облачных провайдеров) и **не** `CONTENT_REJECTED` (модерации нет).
Это локальный сбой среды/входа: битый исходник, нет файла, ошибка кодека, нехватка ресурса.
По природе ближе к `LOCAL_INFERENCE_FAILED` (локально, ресурс/среда), но точный код/обработку
монтажных сбоев финализируем при сборке блока (после подбора обёрток). Принцип тот же: сервер
ловит факт, чинит что может (повтор/проверка входа), спорное/неустранимое — человеку.

## 9. РАЗМЕЩЕНИЕ И СВЯЗИ

Обёртки/адаптер → `core/providers/ffmpeg/`. Тонкие инструменты монтажа → `tools/video/`
(конкретный набор — после подбора обёрток). Раскладки → `workspace/.../scene_layouts/`.
Рендер → `channel_config.RENDER_CONFIG`. Состав сцены → `video_data.SCENES`. Активные типы
фрагментов → `channel_config.SCENE_PROFILE`. Исходники → ИНСТРУКЦИЯ media.
