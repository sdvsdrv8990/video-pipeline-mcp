# БРИФ ДЛЯ ИИ-РАЗРАБОТЧИКА: Слой табличных инструментов (Table Data Layer)

## 0. Как читать этот документ

Это спецификация на реализацию **одного Bounded Context** — слоя работы с табличными данными MCP-сервера. Реализуй строго по этому брифу. Если что-то в брифе противоречит твоему «привычному» решению — следуй брифу: архитектура продумана заранее и каждое ограничение здесь намеренное.

Главное правило, которое нельзя нарушать ни в одном месте: **поведение задаётся конфигом (схемой таблицы), а не Python-кодом.** В коде не должно быть ни одной ветки вида `if table == "video_data"`. Один generic-движок обслуживает все таблицы; различия между таблицами живут только в их схемах.

---

## 1. Контекст системы (зачем это нужно)

Сервер — исполнитель и валидатор. Оркестратор — внешний ИИ, подключённый через туннель. ИИ читает данные, принимает решения и шлёт серверу **намерения**, а не готовые структуры данных и не код.

Существующие принципы системы, которые этот слой обязан соблюдать:

- **Никакого хардкода.** Любое значение, порог, имя столбца — из конфига, не из кода.
- **Excel = Source of Truth.** ИИ никогда не читает и не пишет Excel напрямую. Сервер делает снапшоты Excel в JSON-обёртки и применяет изменения через очередь.
- **JSON-обёртки = интерфейс для ИИ.** ИИ парсит только нужные ему данные, не загружая таблицу целиком (экономия токенов).
- **Строгая валидация.** Каждый запрос проходит Pydantic-валидацию. Любой инструмент возвращает чёткий статус: `success` либо `error` с человекочитаемым описанием причины.

Этот слой расширяет уже существующие инструменты Категории 3 (`json_read_snapshot`, `json_push_to_queue`, `json_execute_queue`, `json_clear_queue`) — он становится их **внутренним движком**. См. раздел 9 про стыковку.

## 2. Модель данных (самое важное — прочитать дважды)

### 2.1. Единица хранения — СТРОКА, адресуемая по ID. Не столбец.

Под капотом каждая таблица хранится как объект, где ключ — это ID строки, а значение — вся строка целиком:

```json
{
  "table": "video_data__META",
  "schema": {
    "video_id":  { "writable": false, "computed": false, "type": "string", "role": "id" },
    "title":     { "writable": true,  "computed": false, "type": "string", "max_len": 100 },
    "channel_id":{ "writable": false, "computed": false, "type": "string", "role": "fk" },
    "status":    { "writable": true,  "computed": false, "type": "enum",
                   "enum": ["IDEA","DRAFT","IN_PRODUCTION","RENDERING","PUBLISHED","ARCHIVED"] },
    "duration_sec": { "writable": true, "computed": false, "type": "integer" }
  },
  "rows": {
    "VID_fin_a1b2": { "title": "...", "channel_id": "CH_fin_x1", "status": "DRAFT",  "duration_sec": 0 },
    "VID_fin_c3d4": { "title": "...", "channel_id": "CH_fin_x1", "status": "PUBLISHED", "duration_sec": 312 }
  }
}
```

**Почему именно так:** если бы единицей хранения был столбец (плоский массив), строки держались бы вместе только позицией в массиве. Прерванная или конкурентная запись → столбцы «разъезжаются» (torn write): пятая строка `status` перестаёт соответствовать пятой строке `title`, файл при этом валиден, и перекос замечается через недели по сломанным связкам. Привязка к ID убирает этот класс ошибок полностью — ничего не выровнено по индексу, всё привязано к ключу.

### 2.2. «Столбец» — это ВИД (проекция) поверх строк, способ чтения

ИИ по-прежнему может тянуть один столбец и экономить токены. Но это вычисляется сервером поверх row-хранилища, а не хранится отдельно:

`get_column(table, "status")` → 

```json
{ "VID_fin_a1b2": "DRAFT", "VID_fin_c3d4": "PUBLISHED" }
```

Эргономика для ИИ ровно та, что нужна (читает по столбику), но рассинхрон невозможен by design.

### 2.3. Запись — всегда read-modify-write ПО СТРОКЕ

Даже когда меняется одно поле, операция внутри сервера = взять строку по ID целиком → поменять поле → положить строку целиком обратно. Никогда не «дописать в столбец». Снаружи для ИИ это выглядит как «поменял одно поле», вся работа со строкой скрыта в сервере.

## 3. Схема таблицы (`schema.yaml`) — статичный конфиг, его задаёт сервер, не ИИ

На каждую таблицу — один файл схемы. Схема — **источник истины о структуре**: какие столбцы есть, какие writable, какие computed, типы, enum-значения, какой столбец является ID, какие — внешними ключами (fk).

Обязательные флаги на каждый столбец:

- `writable` — может ли ИИ писать в этот столбец вообще.
- `computed` — является ли столбец расчётным (Excel-формула). **Запись в `computed: true` сервер ОБЯЗАН отклонять с ошибкой**, а не молча затирать формулу. Это критично: половина таблиц содержит формулы (`retention_percent`, `like_rate`, `uniqueness_score`, `freshness_score`, `performance_score` и т.д.).
- `type` — для типовой валидации (запись строки в integer-поле → отказ).
- `role` — опционально: `id` (первичный ключ строки) или `fk` (внешний ключ на другую таблицу).

ИИ **никогда не сочиняет и не редактирует схему.** Он её только читает (через отдельную read-операцию, если нужно), чтобы понять, что доступно.

### 3.1. Интроспектор схем (обязательный вспомогательный скрипт)

Поскольку столбцы во всех таблицах уже существуют в Excel, напиши скрипт-интроспектор, который проходит по книге, читает заголовки столбцов и **автогенерирует черновики схем для всех таблиц**. Правила автодетекта:

- Столбец, значение которого в Excel начинается с `=`, помечается `computed: true, writable: false`.
- Столбец с пометкой «Read-Only» / «Read-Only (Formula)» в исходной документации → `writable: false`.
- Тип определяется по данным/документации (string/integer/float/enum/date).

Человек после генерации только проверяет и доставляет недостающие `computed`-флаги. Это полдня работы вместо ручного написания десятков схем.

## 4. Пять примитивов — единый интерфейс для ИИ на ВСЕ таблицы

ИИ оперирует ровно пятью глаголами. Куда они применяются — задаётся параметром `table`. Новых операций «под таблицу» быть не должно.

### Чтение (могут идти параллельно, без блокировок)

**`table_get_column(table, column)`**
Возвращает проекцию одного столбца: `{ row_id: value }`. Экономия токенов.

**`table_get_row(table, row_id)`**
Возвращает одну строку целиком: `{ column: value }`.

(Опционально, если нужно: `table_get_rows(table, row_ids[])` и `table_query(table, filter)` — но это надстройки поверх get_row, не отдельная логика хранения.)

### Запись (идёт ТОЛЬКО через очередь, сериализуется на таблицу — см. раздел 5)

**`table_set(table, row_id, column, value)`**
Изменить одно поле в существующей строке. Внутри — read-modify-write по строке.

**`table_append(table, data)`**
Добавить новую строку. **ID присваивает СЕРВЕР, не ИИ** (конкретные форматы ID — в
отдельной сессии по реестру ID; ключ ассета — `global_asset_id`). Новая строка «падает
ниже» с новым ID — поведение как у Airtable/Excel, но детерминированное.

**`table_delete(table, row_id)`**
Удалить строку по ID (с флагом подтверждения, по аналогии с `fs_delete`).

## 5. Очередь записи — по одной на таблицу, запись сериализуется

Это решает **конкурентную запись** (два процесса пишут в одну таблицу одновременно), а не только rate limit. Правила:

- **Одна последовательная очередь записи на каждую таблицу.** Не одна общая на весь сервер (это убьёт параллелизм — пока пишется `video_data`, должна спокойно писаться `channel_stats`). И не «пиши когда хочешь» (это гонка).
- **Чтения параллельны и не блокируются.** Читатели друг другу не мешают; мешает только запись, и только в пределах своей таблицы.
- Каждая операция записи (`set`/`append`/`delete`) кладётся в `pending_updates` соответствующей `write.json`. Сервер применяет очередь атомарно к Excel, обновляет снапшот (`read.json`) и триггерит Qdrant (для приоритетных данных) — это уже существующее поведение `json_execute_queue`, сохрани его.

---

## 6. Межтабличная связь — только через ID как join-ключ, без общих буферов

Когда инструмент одной таблицы должен обновить другую (пример: инструмент видео обновляет статистику канала), он **не лезет в данные чужой таблицы напрямую**. Он кладёт операцию в очередь записи целевой таблицы, адресуя строку **по ID** (`channel_id`), а не по позиции.

Это реализует существующую ДОГМУ 2 (хлебные крошки): связи `asset_id → scene_id → channel_id` живут через ID и не рвутся, даже когда пишут разные инструменты.

---

## 7. Контракт инструмента — кто какие столбцы вправе трогать

Схема таблицы (`writable`/`computed`) защищает данные, но **не ограничивает, какой инструмент к каким столбцам обращается.** Это задаётся в контракте каждого инструмента-потребителя (вне этого слоя, но слой обязан это поддерживать): инструмент декларирует, какие столбцы читает и какие пишет, и сервер отклоняет попытку выйти за рамки контракта.

Слой данных должен принимать опциональный параметр «контекст вызывающего» и уметь применять это ограничение, если оно передано. Без него god-object вернётся через чёрный ход: данные разрезаны, но лазить по ним сможет кто угодно.

## 8. ЧЕГО ДЕЛАТЬ НЕЛЬЗЯ (анти-паттерны — каждый ломает архитектуру)

1. **Не делать master read/write JSON**, агрегирующий все таблицы в один файл. Он раздувает токены, возвращает гонку записи и рушит изоляцию. Это god-object в формате JSON. Допустим только тонкий read-only **индекс существующих ID** (без бизнес-данных), и то опционально.
2. **Не давать ИИ формировать структуру** конвертов или схем. ИИ шлёт только намерения (5 примитивов). Структуру лепит сервер.
3. **Не хранить данные по столбцам / по позиции в массиве.** Только строки по ID.
4. **Не писать в столбец напрямую** в обход read-modify-write строки.
5. **Не затирать `computed`-столбцы.** Запись в них → немедленный отказ с объяснением.
6. **Не присваивать ID на стороне ИИ.** ID раздаёт только сервер.
7. **Не ветвить код по имени таблицы.** Один generic-движок + схемы-конфиги.

## 9. Стыковка с существующими инструментами Категории 3

Существующие `json_*` инструменты остаются публичным интерфейсом, этот слой — их движок:

- `json_read_snapshot` — отдаёт ИИ конверт `read.json`, сформированный сервером из row-хранилища (с поддержкой проекций столбцов через `table_get_column`).
- `json_push_to_queue` — кладёт намерение в `pending_updates`. Семантические `action` (`update_performance`, `add_technique` и т.п.), которые уже есть в системе, должны транслироваться в низкоуровневые примитивы (`set`/`append`) этого слоя. То есть семантические действия — это сахар поверх пяти примитивов, а не отдельная логика хранения.
- `json_execute_queue` — применяет очередь через row-based read-modify-write, обновляет снапшот и Qdrant.
- `json_clear_queue` — без изменений.

Сохрани правило Missing Component Protocol: попытка операции над несуществующим файлом/таблицей → отказ с инструкцией создать структуру (`fs_create_file` / `fs_create_project_structure`), а не молчаливая ошибка.

## 11. Критерии приёмки (тест-сценарии на реальной таблице)

Возьми реальную таблицу `video_data` (листы `META`, `PERFORMANCE`). Реализация принята, если:

1. **Чтение столбца.** `table_get_column("video_data__META", "status")` возвращает `{ id: value }` только по статусам, без остальных полей.
2. **Изменение поля.** `table_set("video_data__META", "VID_fin_a1b2", "status", "RENDERING")` меняет только это поле; остальные поля строки и все прочие строки не затронуты.
3. **Добавление строки.** `table_append("video_data__META", {title:"X", channel_id:"CH_fin_x1"})` создаёт строку с **серверным** ID; существующие строки на месте.
4. **Защита формул.** `table_set(..., "PERFORMANCE", row, "retention_percent", 50)` → **отказ** (`computed: true`), формула цела.
5. **Типовая валидация.** Запись строки в `duration_sec` (integer) → отказ с понятной ошибкой.
6. **Конкурентная запись.** Две одновременные записи в `video_data` применяются последовательно без перекоса столбцов; параллельное чтение `channel_stats` при этом не блокируется.
7. **Межтабличность.** Инструмент видео роняет апдейт в очередь `channel_stats` по `channel_id`; связь по ID не рвётся.
8. **Никакого хардкода.** Добавление новой таблицы = добавить её `schema.yaml`; код движка не меняется.

---

## 12. Резюме одной фразой

Один generic-движок хранит данные строками по ID, отдаёт их ИИ проекциями столбцов, пишет атомарно через очередь-на-таблицу, валидирует всё по декларативным схемам и защищает формулы — а ИИ управляет этим пятью глаголами, не зная и не формируя ни одной внутренней структуры.
