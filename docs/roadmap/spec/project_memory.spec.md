# СПЕКА: `project_memory.md` — память проекта

> Формат файла памяти. Это **данные**, живут в `workspace/` рядом с каналом и видео;
> ведёт state-слой (`core/state/`). Не путать с логом и не дублировать таблицы.

---

## 0. ЧТО ЭТО И ЧЕМ ОТЛИЧАЕТСЯ ОТ ЛОГА И ТАБЛИЦ

Три разных хранилища, три разных вопроса. Память НЕ заменяет ни одно:

| Хранилище | Вопрос | Кто читает |
|---|---|---|
| `_SESSION_LOG` (лог) | что ПРОИЗОШЛО (события, хронология) | машина |
| Таблицы (.xlsx) | какое СОСТОЯНИЕ и МЕТРИКИ (числа, статусы) | сервер/Claude |
| `project_memory.md` (память) | что РЕШИЛИ и ЧТО ИЗ ЭТОГО ВЫШЛО | человек/Claude |

Лог — поток сырых событий. Таблица — текущие данные. Память — **осмысленные решения
с последствиями**, история работы проекта простым языком.

---

## 1. ЖЕЛЕЗНОЕ ПРАВИЛО: НЕ ДУБЛИРОВАТЬ ТАБЛИЦЫ

Память — частая ловушка: превращается в кривую копию таблиц, два источника истины
расходятся, всё гниёт. Поэтому граница жёсткая:

- **Число или статус** → это в ТАБЛИЦЕ, в память НЕ копируется.
  Не «уникальность стала 0.82» — это в таблице.
- **Причина решения и его эффект** → это в ПАМЯТИ, в таблице такого нет.
  «Решили добавить вариацию AST_x, потому что приелся; результат — см. динамику в таблице».

Память **ссылается** на таблицы по ID, не дублирует значения. Лакмус: если данные
можно прочитать из таблицы — память на них только ссылается, не повторяет.

## 2. ДВА УРОВНЯ (по иерархии, как таблицы)

| Уровень | Файл | Что хранит |
|---|---|---|
| **Видео** | `…/videos/{video}/project_memory.md` | решения по ЭТОМУ видео (правки сценария, выбор ассетов, спорные моменты в его производстве) |
| **Канал** | `…/channels/{channel}/project_memory.md` | кросс-видеовые решения и итоги (стратегия, ротация, что сработало по каналу в целом) |

Связь как у таблиц: видео-память — про конкретное видео; канал-память — про то, что
видно только сверху, по всем видео. Видео-решение может подытоживаться в канал-память,
когда влияет на стратегию (по аналогии с агрегацией метрик).

## 3. ЗАМКНУТАЯ ПЕТЛЯ: сигнал → решение → флаг → результат

Главная ценность памяти — отслеживать, **что на что повлияло, по фактам, а не по
догадкам ИИ**. Петля:

```
1. скрипт-сверка поймал сигнал           → факт в таблице (UNIQUENESS_ALERTS)
2. спорно → решение человека             → ЗАПИСЬ в память: что решили + ПОЧЕМУ
3. Claude ставит флаг на сигнал          → signal_status + flag_source=human_decision
   (флаг ВЫТЕКАЕТ из решения человека, не из мнения ИИ)
4. позже: РЕЗУЛЬТАТ дописывается в память → сработало/нет, ссылка на эффект по ID
5. петля замкнута: видно "такое решение в такой ситуации дало такой эффект"
```

Ключевое: запись решения **без результата неполна**. Результат дописывается потом, по
фактам из таблиц (по ID), а не по ощущению Claude. Так копится знание на фактах.

---

## 4. СТРУКТУРА ЗАПИСИ (рекомендуемая)

Каждая запись — это решение со своей судьбой:

```markdown
## [дата] Решение: <короткий заголовок>
- **Контекст:** что произошло (ссылка на сигнал/ID, не копия цифр)
- **Кто решил:** человек / Claude (с подтверждением человека)
- **Решение:** что именно сделали
- **Почему:** причина (этого нет в таблицах)
- **Результат:** [дописывается позже] эффект по фактам (ссылка на ID/динамику)
```

Поля `Кто решил` и `Почему` — то, чего нет нигде в таблицах, ради них память и нужна.
`Результат` сначала пустой, заполняется когда эффект проявился.

---

## 5. КАК ПИШЕТСЯ (инструменты)

Память — **файл**, не таблица: пишется `fs_*` (`fs_read_file` / `fs_write_file` с
осторожностью, обычно дозапись новой записи). НЕ через очередь таблиц. Флаг сигнала,
на который ссылается запись, живёт в таблице (`signal_status`, `flag_source`) и
меняется `table_set`. То есть: решение и причина → в память (файл), флаг состояния
сигнала → в таблицу (данные). Память и флаг связаны по ID сигнала, не дублируют друг друга.

---

## 6. ЧЕГО В ПАМЯТИ НЕ ДОЛЖНО БЫТЬ

- Чисел и статусов, которые есть в таблицах (только ссылки по ID).
- Сырых событий (это `_SESSION_LOG`).
- Флага как такового (он в таблице; в памяти — решение, из которого флаг вытек).
- Догадок Claude без пометки, что это догадка, а не решение человека или факт.
