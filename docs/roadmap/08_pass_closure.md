## 0. Принцип: GIT-NATIVE история (владелец 2026-07-05)

**Git И ЕСТЬ история версий — файлов И кода — и это лучше текстовых `history_*.md`.** Используем на полную:
- **История файла** = `git log --follow <file>` · `git blame` · `git show <commit>` · `git diff` (не рукописный `history_*.md`, который гниёт и дублирует).
- **Решение→факт (почему)** = **commit-сообщение** (тело: что/почему/возможные регрессии/`F#`). Атомарные коммиты + дисциплина сообщений = теперь ЧАСТЬ процесса, не опция.
- **Нарратив сессии** (много файлов) = `_sessions.md`. **Кросс-сессия/резюме/решения владельца** (чего в git нет) = память.
- **Обзор/ревью** = PR + `git blame`; **версии** = tags.

Следствие: `history_*.md` больше НЕ ведём. Скилы перепрошиты git-native баннером (все 9). Авторитет процесса = память `project-workflow-canonical`.

## 6. Честно закрытый проход — Definition-of-Closed (владелец, Сессия 15)

Проход (гейт зрелости ИЛИ ревью-прогон §3.1) считается **ЗАКРЫТЫМ** только когда пройдены все три
ступени по ДОКАЗАННОМУ состоянию. Порядок жёсткий: обмер → адаптация → двойное подтверждение.

**Ступень A — полный обмер (не выборка).**
- Пройтись по **ВСЕМ файлам** зоны прохода, выявить **ВСЕ возможные проблемы**: архитектурные
  аномалии + прочие нарушения паттернов (словарь `D#/G#`, скилы `DIM-13…16`).
- Для каждой затронутой **системы** зафиксировать **«что есть сейчас» против «что должно быть»**
  на серьёзной зрелости (целевые уровни `07`).
- Всё найденное → `02_findings.md` (F#), с привязкой к DIM.

**Ступень B — готовые решения с GitHub (адаптировать, где возможно и необходимо).**
- Подобрать сильное готовое решение (§4 / `04_github_adoption.md`) и уложить под нас **правилом
  адаптации §4** (через `project-conventions` + `anti-hardcode` + `security-reviewer`, не «скопировать репо»).
- **ИСКЛЮЧЕНИЕ — БЕЗОПАСНОСТЬ:** паттерны угроз и механизмы фиксов **уже собраны** (`06` IN/OUT/§F/§G/§H
  + `patterns.yaml`). Повторно искать/адаптировать с GitHub по безопасности **НЕ нужно** — берём из `06`.

**Ступень C — двухступенчатое тестовое подтверждение (обе обязательны).**
- **C1 — простые тесты** (`tests/quick/`, unit/contract): подтверждение теории **ЧЕРЕЗ КОД** —
  гипотеза находки/фикса воспроизведена и закрыта на уровне кода + контракта/реакции.
- **C2 — тесты-симуляции** (`tests/*_injection`, `render_*`, `agent_swarm`): **ФИНАЛЬНОЕ**
  подтверждение — используют **РЕАЛЬНЫЕ инструменты** против живого сервера.
  - **Требование C2 (владелец):** симуляциям нужен **доступ на ЧТЕНИЕ КОНСОЛИ** (stdout/логи
    сервера) — иначе реальный эффект не наблюдаем и подтверждение неполно. Это capability
    тест-харнесса (стык `03_testing_plan.md`; пересекается с DIM-7, но здесь — как условие закрытия).
- Проход закрыт **ТОЛЬКО** когда C1 **и** C2 зелёные: C1 без C2 = теория без реальности; C2 без
  C1 = эффект без доказанной причины.

> Формула: **обмер всех файлов → now-vs-target → адаптировать GitHub (кроме sec) → простые тесты
> (код-пруф) → симуляции с чтением консоли (реал-пруф) → проход закрыт.**
>
> Связь с зрелостью: проход поднимает свои DIM (§3.1) в `07` **только** после ступени C. До неё —
> уровень остаётся прежним, даже если «работа сделана».
