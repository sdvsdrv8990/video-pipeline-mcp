# Рабочий процесс v2

> **Важно:** перед любой задачей пройти `CONTEXT_PROTOCOL.md` — определить тип задачи, получить список файлов для чтения, собрать контекст, честно ответить на вопросы из `questions.md`. Без контекста — код будет дублировать или ломать прошлое.

## Иерархия документов

```
audit/v2/README.md          ← операционный мануал (СТАРТ ЗДЕСЬ)
audit/v2/SESSIONS.md        ← хронология: сессия → решения → D# → G#
audit/v2/AUDIT.md           ← что улучшить: дефекты D# с эмприкой
audit/v2/global.md          ← почему так: сквозные решения G#
audit/v2/files/*.md         ← почему так: файловые разборы Q&A
audit/v2/files/github_references.md ← паттерны из GitHub для каждого модуля

docs/dev/*.md               ← детальные спецификации (чтение после v2)
docs/dev/history_*.md       ← эволюция решений в файлах (заполняется при правке кода)
docs/dev/threat_landscape.md← паттерны атак (читать перед security-задачами)
docs/dev/server_reactions.md← реестр реакций (читать перед ошибками)
```

**Правило:** v2 = авторитет. Если исходный файл противоречит AUDIT.md/global.md — побеждает v2.

**Ограничение:** НОВЫЕ ПРОВАЙДЕРЫ НЕ ДОБАВЛЯЮТСЯ. У нас в заглушках уже есть всё для TTS/STT/IMG/FFmpeg. Референсы используются ТОЛЬКО для паттернов и архитектурных решений.

## Типы задач

### 1. Новый код
```
1. SESSIONS.md — что уже решали похожее
2. AUDIT.md — не повторяем ли D#?
3. files/<файл>.md — решения этого файла
4. global.md — сквозные G#
5. github_references.md — паттерны для модуля (если есть)
6. 5 вопросов + 4 уровня анализа
7. Код → history_<модуль>.md → SESSIONS.md (обновить если новое решение)
```

### 2. Исправление ошибки
```
1. SESSIONS.md — ищем похожие баги
2. AUDIT.md — связан ли с существующим D#?
3. server_reactions.md — какой код ошибки?
4. github_references.md — паттерны исправления для модуля
5. Root cause → исправление → REGRESSION TEST
6. history_<модуль>.md → SESSIONS.md + AUDIT.md (обновить)
```

### 3. Рефакторинг
```
1. SESSIONS.md — почему код устроен так
2. files/<файл>.md — какие альтернативы отброшены
3. global.md — не ломаем ли G#?
4. github_references.md — лучшие паттерны для модуля
5. Не отменяем решения без веской причины
6. history_<модуль>.md → SESSIONS.md (записать почему пересмотрели)
```

### 4. Security-аудит
```
1. threat_landscape.md — какие атаки актуальны
2. AUDIT.md — открытые security D# (D14, D29, D31)
3. firewall_sessions.md — план развития
4. global.md — G3, G12, G17, G18
5. files/core_firewall_*.md — разбор правил
6. github_references.md — паттерны безопасности
7. Эмпирика: запуск на .venv, реальные payload'ы
```

### 5. Добавление инструмента
```
1. SESSIONS.md §Сессия 3 — как добавлялись прошлые
2. architecture.md — три фазы, контракты
3. server_reactions.md — нужен ли новый код ошибки?
4. github_references.md — паттерны для типа инструмента
5. templates.md — шаблон, если затрагивает workspace
6. 5 вопросов + 4 уровня → код → history_<модуль>.md → SESSIONS.md
```

### 6. Тестирование
```
1. testing_strategy.md — стратегия тестирования
2. 5 вопросов перед тестом
3. Выбрать тип: быстрый (tests/quick/) или постоянный (tests/<name>/)
4. Написать тест → запустить → проверить
5. Если постоянный — добавить history.md в папку теста
```

### 7. Добавление нового инструмента (полный цикл)

**Шаг 1: Определить необходимость**
```
- Есть ли уже такой инструмент? (проверить tools/, server.py)
- Какой Bounded Context? (filesystem/tables/excel/media/video)
- Нужен ли новый код ошибки? (проверить server_reactions.yaml)
```

**Шаг 2: Спроектировать**
```
- Имя: <контекст>_<действие> (например fs_write_file)
- Вход: input_schema (Pydantic/jsonschema)
- Выход: ToolResult{status, data, error, facts}
- Containment: safe_resolve для путей
- Реакция: код ошибки из реестра или новый
```

**Шаг 3: Реализовать**
```
- Хендлер: async def, возвращает ToolResult
- Валидация: input_schema + safe_resolve (для путей)
- Ошибки: ErrorDetail с кодом из реестра
- Facts: Fact(type=..., data={...})
- Регистрация: engine.register(name, description, input_schema, handler)
```

**Шаг 4: Протестировать**
```
- Happy path: валидные параметры → success
- Edge cases: пустые поля, длинные строки, спецсимволы
- Ошибки: невалидные параметры → VALIDATION_ERROR
- Containment: traversal → PATH_ESCAPE
- Facts: проверить что facts создаются
```

**Шаг 5: Задокументировать**
```
- history_<модуль>.md: vX.Y — дата — что сделано
- files/<файл>.md: обновить если есть
- SESSIONS.md: добавить в текущую сессию
- AUDIT.md: обновить покрытие
```

| Уровень | Вопрос | Пример |
|---|---|---|

## История файлов (обязательно при правке кода)

При КАЖДОМ изменении исходного кода (*.py) — обновить соответствующий `docs/dev/history_<модуль>.md`:

| Что трогаешь | История |
|---|---|
| `server.py` | `docs/dev/history_server.md` |
| `core/contracts/*` | `docs/dev/history_core_contracts.md` |
| `core/engine/*` | `docs/dev/history_core_engine.md` |
| `core/transport/*` | `docs/dev/history_core_transport.md` |
| `core/firewall/*` | `docs/dev/history_core_firewall.md` |
| `core/state/*` | `docs/dev/history_core_state.md` |
| `core/reactions/*` | `docs/dev/history_core_reactions.md` |
| `core/ids/*` | `docs/dev/history_core_ids.md` |
| `core/providers/*` | `docs/dev/history_core_providers_<тип>.md` (tts/stt/img/ffmpeg) |
| `tools/*` | `docs/dev/history_tools_<контекст>.md` (filesystem/tables/excel/media/video) |
| `pipeline/*` | `docs/dev/history_pipeline.md` |
| `config/*` | `docs/dev/history_config.md` (только если меняется логика, не значения) |

**Формат записи (новая версия):**

```markdown
## vX.Y — <дата> — <заголовок изменения>

### Решение
- Что сделано и почему (с привязкой к D#/G#)

### Регрессия
- Что может сломаться, где проявится в системе

### Связь
- Ссылки на файлы/решения/D#/G#
```

**Правило:** история файлов = эволюция решений. Не переписывать прошлое, только добавлять новые записи. Если история файла не существует — создать с текущей записью. Если история существует — добавить запись в конец.

**Write-back включает:** `SESSIONS.md` + `AUDIT.md` + `global.md` + `files/<файл>.md` + `history_<модуль>.md`.

## GitHub референсы — как работать с референсами

**Полный каталог:** `audit/v2/files/github_references.md` — 20+ репозиториев по модулям.

### Важно: НОВЫЕ ПРОВАЙДЕРЫ НЕ ДОБАВЛЯЮТСЯ

У нас в заглушках (`core/providers/`) уже есть ВСЁ для работы с контентом:
- `tts/` — TTS через LiteLLM (облачный)
- `stt/` — STT через stable-ts (локальный)
- `img/` — IMG через LiteLLM (облачный)
- `ffmpeg/` — FFmpeg через внешний MCP-сервер

**Референсы используются ТОЛЬКО для:**
1. Паттернов и архитектурных решений (как组织ировать код)
2. Исправления дефектов (как чинить D#)
3. Тестирования (как тестировать MCP-серверы)
4. Безопасности (как защитить firewall/auth/state)

**НЕ для:**
- Добавления новых провайдеров (уже есть)
- Замены существующих провайдеров (LiteLLM/stable-ts — наш выбор)
- Копирования кода 1:1 (адаптировать паттерн, не код)

### Порядок работы с референсами при задаче

```
1. Определи модуль задачи (core/tools/pipeline/config)
2. Открой github_references.md → найди модуль
3. Прочитай референсы модуля (таблица + решение)
4. Извлеки паттерн (НЕ код):
   - Как组织изован код?
   - Какие контракты используются?
   - Какие паттерны безопасности?
   - Какие паттерны тестирования?
5. Прогони паттерн через 5 вопросов + 4 уровня
6. Адаптируй под нашу архитектуру (ToolResult, Engine, Firewall)
7. Реализуй с учётом D#/G#
```

## Как делать аудит файлов

### Принцип: один файл = один проход

Не нужно создавать кучу файлов на каждый модуль. Один файл аудита содержит ВСЁ: Q&A, дефекты, решения, связи. Просто нужно сделать несколько проходов по этому файлу.

### Формат одного файла аудита

```markdown
# Аудит: <имя модуля/файла>

## Суть (одна строка)
Что делает этот файл, зачем он нужен.

## Текущее состояние
- Статус: реализован / заглушка / частично
- Строки: XXX
- Зависимости: [список импортов]

## Проход 1: Понимание (что есть)
### Q: Зачем существует?
A: [Ответ на инженерный вопрос]

### Q: Как работает?
A: [Поток данных, входы/выходы]

### Q: Какие контракты?
A: [ToolResult, ErrorDetail, Fact, и т.д.]

### Найденные проблемы:
| # | Проблема | Серьёзность | Доказательство |
|---|---|---|---|
| 1 | [описание] | 🟡/🟠/🔴 | [запуск/grep/логика] |

## Проход 2: Исправление (что меняем)
### Q: Как чиним?
A: [Конкретный план с привязкой к D#/G#]

### Q: Что может сломаться?
A: [Регрессия]

### Q: Как тестировать?
A: [Конкретные тесты]

### Закрытые проблемы:
| # | Проблема | Решение | Доказательство |
|---|---|---|---|
| 1 | [описание] | [фикс] | [тест] |

## Проход 3: Интеграция (как связать)
### Q: Как влияет на систему?
A: [Поток данных через модуль]

### Q: Какие G# затронуты?
A: [Ссылки на global.md]

### Q: Что записать в историю?
A: [Запись в history_*.md]
```

### Как выбирать вопросы для аудита

**Не все вопросы подходят для каждого файла.** Выбирай по типу модуля:

| Тип модуля | Какие вопросы | Почему |
|---|---|---|
| **Контракты** (contracts/) | Код, поведение, поток данных, долгосрочный | Контракты = закон, их нельзя менять без анализа |
| **Движок** (engine/) | Поведение, поток данных, регрессия | Движок = сердце, ошибки каскадом |
| **Firewall** (firewall/) | Безопасность, отказоустойчивость, доказательства | Firewall = барьер, должен работать всегда |
| **Транспорт** (transport/) | Протокол, совместимость, безопасность | Транспорт = мост, ломается = всё ломается |
| **Инструменты** (tools/) | Контракты, валидация, containment | Инструменты = лицо, Claude их видит |
| **Конфиги** (config/) | Валидация, дефолты, миграция | Конфиги = данные, неправильный = катастрофа |
| **Тесты** (tests/) | Покрытие, изоляция, воспроизводимость | Тесты = доверие, сломались = нет доверия |

### Формула выбора вопросов

```
1. Определи тип модуля (контракты/движок/firewall/транспорт/инструменты/конфиг/тесты)
2. Возьми вопросы из таблицы выше
3. Добавь универсальные:
   - Что читаю перед правкой?
   - Что может сломаться?
   - Как доказать что работает?
4. Ответь на каждый вопрос (честно, с доказательствами)
5. Запиши найденные проблемы в таблицу
```

### Пример: аудит core/contracts/error_detail.py

**Проход 1: Понимание**
```
Q: Зачем существует?
A: Обёртка ошибки для Claude. Содержит код, сообщение, recovery, raw_response.

Q: Как работает?
A: Pydantic модель. Claude получает при ЛЮБОЙ ошибке. Код → маппинг на server_reactions.yaml.

Q: Какие контракты?
A: ErrorDetail используется в ToolResult.error, TaskStatus.error, Engine._error.

Найденные проблемы:
| # | Проблема | Серьёзность | Доказательство |
```
