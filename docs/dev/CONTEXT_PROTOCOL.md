# Протокол сбора контекста

**Назначение:** при ЛЮБОЙ задаче по коду (создание, баг-фикс, рефакторинг, security) ИИ обязан пройти протокол, чтобы **честно ответить** на каждый вопрос из `questions.md`. Пропуск шага = ложный ответ.

**Принцип:** контекст = знание того, что **уже решено** (SESSIONS/files/global) + что **сломано** (AUDIT) + что **нужно сделать** (questions). Без этого — код будет дублировать или ломать прошлое.

---

## 0. Определение типа задачи

**Перед чтением** определить тип и получить список секций вопросов:

| Тип задачи | Применяются секции вопросов | Ключевые файлы |
|---|---|---|
| **Новый модуль** | A1–A6, C1–C17, D1–D12, Q1–Q6 | architecture, core, contracts, templates |
| **Баг-фикс** | R1–R6, C6–C16, Q1–Q6 | AUDIT, SESSIONS, relevant files |
| **Security-задача** | S1–S20, A1–A6, Q1–Q6 | threat_landscape, AUDIT, firewall, global |
| **Добавление инструмента** | A1–A6, C1–C17, D1–D12, R1–R6 | architecture, core, server, tools |
| **Рефакторинг** | Все секции, затронутые файлом | SESSIONS, files/*.md, global, AUDIT |
| **Тестирование** | Q1–Q6 + секции тестируемого модуля | testing_strategy, relevant files |
| **Пайплайн** | D1–D12, A1–A6, T1–T9 | pipeline, templates, server |
| **Конфиги** | D10–D12, T1–T9, I1–I7 | config/*, server, tunnel |

## Общая формула: сколько проходов нужно

| Тип задачи | Минимум проходов | Файлов за проход | Общее время |
|---|---|---|---|
| Тривиальный баг | 1 (только §соответствующее) | 3–5 | 5 мин |
| Новый инструмент | 2 (сквозное + детали) | 8–12 | 15 мин |
| Security-задача | 3 (сквозное + файрвол + транспорт) | 15–17 | 30 мин |
| Рефакторинг ядра | 3 (сквозное + контракты + файлы) | 12–15 | 25 мин |
| Архитектурное решение | 4 (все секции) | 20+ | 45 мин |

**Правило:** чем больше blast-radius изменения — тем больше проходов. Изменение в одном файле = 1–2 прохода. Изменение сквозного решения = все 10 секций.
