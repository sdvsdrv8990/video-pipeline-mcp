# Глобальный разбор системы — сквозные решения (v2)

Здесь — решения, действующие во **всём** проекте, не принадлежащие одному файлу. Файловые разборы (`files/*`) ссылаются сюда по якорям (`global.md#g3`), а не переписывают текст. Дефекты — в [`AUDIT.md`](AUDIT.md) (`D#`); здесь только «почему так».

Обозначения: **Q** — инженерный вопрос, **A** — принятое решение и причина, **Alt** — отброшенная альтернатива, **Регрессия** — что сломается при смене решения.

Помечены `[v1→v2 pending]`.

> **S2 пройден (2026-07-03):** новые [G14](#g14-внутренний-toolresult-богаче-того-что-доходит-до-claude-на-проводе) (ToolResult богаче провода — зонтик D20/D24/D27/D30), [G15](#g15-строковые-словари-без-единого-реестра-дрейфят-literalреестр--лекарство) (дрейф словарей D4/D25 vs эталон), [G16](#g16-незавершённость-должна-кричать-честная-заглушка-а-не-молчать) (незавершённость должна кричать D24/D3/D28) — доказаны запуском. **S3 пройден (2026-07-03):** новые [G17](#g17-containment-workspace--единая-точка-а-не-проверка-в-каждом-хендлере) (containment = choke-point, D29/D1), [G18](#g18-за-туннелем-клиент-один--гранулярность-по-ip-бессмысленна-секрет-уязвим) (за туннелем клиент один → IP≠auth, D14/D3/D31). **S6 (2026-07-03):** D3 закрыт (bearer auth). **ОСТАЛОСЬ (финализация F):** актуализировать перенесённые G1–G11 `[v1→v2 pending]` + per-area scores в AUDIT.md.

## G1. Роль сервера: исполнитель+валидатор, оркестратор — снаружи
**Q:** кто принимает решения — сервер или ИИ?
**A:** сервер только исполняет и валидирует; думает внешний Claude через туннель. Сервер не self-healing: ошибку ловит, оборачивает в `ErrorDetail`, отдаёт Claude — он решает retry/смена/человек.
**Alt:** «умный» сервер с автопочинкой — отброшено: дублирует интеллект оркестратора, прячет причины сбоев.
**Регрессия:** если сервер начнёт сам чинить — Claude теряет видимость причин.
**Где:** `server.py`, все `providers/*` (`_map_error`), `reactions`.

## G2. Единый конверт ответа `ToolResult`
**Q:** как Claude отличает успех от ошибки, не зная деталей каждого инструмента?
**A:** любой инструмент возвращает один тип `ToolResult{status,data,error,facts}`. Один парсер у Claude, а не N форматов. Совпадает с духом MCP (`CallToolResult`).
**Alt:** свободный `dict` на инструмент — отброшено: нет типизации, Claude гадает формат (баг v1.1).
**Регрессия:** новый `status` кроме `success|error` → ломает парсинг у Claude.
**Где:** `core/contracts/*`, `engine`, `transport`, инструменты. Разбор базы — `files/core_contracts_tool_result.md` (T2, следующий проход).

## G3. Firewall ПЕРЕД ядром
**Q:** где фильтровать входящие запросы?
**A:** отдельный слой `transport → FIREWALL → engine`. Атака не доходит до инструментов; граница чёткая. Порядок правил — дёшево→дорого: IP-blocklist → rate-limit → injection → anomaly (`Firewall.check()`).
**Alt:** проверки внутри ядра/инструментов — отброшено: размывает изоляцию, каждый инструмент дублирует защиту.
**Дифференцированная реакция (уточнено в v2, против «auto_ban всё» из v1):** бан IP — только при повторных нарушениях rate-limit (`should_ban()`, [D6] закрыт); injection → блок ТОЛЬКО текущего запроса, без авто-бана (эвристика ложнит, Claude — доверенная сторона, [D7] закрыт); anomaly → блок текущего запроса без бана.
**Fail-closed (верифицировано v2):** ошибка парсинга тела → HTTP 400, исключение внутри `firewall.check()` → HTTP 403 block, НЕ пропуск (`server.py:359-380`, явные `# D10: fail-closed`). Старый риск fail-open ([D10]) **закрыт**.
**Регрессия:** слишком строгий firewall → ложные блокировки (за туннелем катастрофа, см. [G12]); перестановка правил меняет и стоимость, и семантику (injection до бана); возврат `except: pass` вокруг `check()` вернёт [D10].
**Где:** `core/firewall/*`, вызов `server.py:365-387`. Детали — `files/core_firewall_firewall.md`, `files/core_firewall_contracts.md`.

## G4. Три фазы инструмента (trigger → poll → download)
**Q:** как единообразно работать и с мгновенными, и с долгими операциями?
**A:** общий жизненный цикл. Фаза 2 (поллинг) — только где есть async (`tts_*`, `img_*`, `ffmpeg render`, длинный `stt`). Синхронные `fs_*`, `table_*`, `excel_*` идут Фаза1→Фаза3.
**Alt:** блокирующий вызов до готовности — отброшено: держит соединение, плохо масштабируется.
**Регрессия:** новый `status` в `TaskStatus` → правка всей poll-логики.
**Где:** `core/contracts/task_status.py`, `providers/*`.

## G5. Философия ошибок: ловим и кодируем, не чиним; отдаём ПОЛНЫЙ текст
**Q:** сколько информации об ошибке давать Claude?
**A:** код (маппинг на `server_reactions.yaml`) + **полный** текст провайдера + `recovery`. Полный текст нужен, чтобы Claude отличил модерацию от таймаута от битого файла.
**Alt:** только код — отброшено: теряется причина.
**Три природы медиа-сбоя:** `PROVIDER_FAILED` (сервер: retry+fallback), `CONTENT_REJECTED` (Claude: переформулировать), `LOCAL_INFERENCE_FAILED` (деградация → человек).
**Где:** `core/contracts/error_detail.py`, `reactions`, `providers/*._map_error`. **Дефект [D4] (частично закрыт v2):** реестр проведён в движок (`engine._error`→`reactions.get_error`), но fs_*/table_* хендлеры `server.py` минуют его и дрейфят кодами вне yaml; `code:str` не привязан к реестру. **[D23]:** `raw_response`/`message` уходят без редактирования секретов. *(Смежно: [D20] — firewall тоже несёт машинный `error_code`, но потребитель его отбрасывает.)*

## G6. Порядок кода в файле = граф зависимостей (снизу вверх)
**Q:** в каком порядке располагать определения внутри файла?
**A:** сначала используемое (вспомогательное), потом использующее. Порядок полей модели: идентификатор → содержимое → связи → мета.
**Alt:** алфавит/произвольно — отброшено: теряется причинно-следственная нить.
**Регрессия:** обратный порядок → forward-reference; в контрактах — риск ошибок импорта.
**Где:** канонично `error_detail.py`, `contracts/__init__.py`, `firewall/contracts.py` (порядок `FirewallDecision`→`FirewallRequest`→`FirewallResult`). Тесно связано с [G13].

## G7. Транспорт — JSON-RPC 2.0 на aiohttp
**Q:** какой протокол и сервер под MCP?
**A:** JSON-RPC 2.0 (стандарт MCP), aiohttp (async, минимальный код).
**Alt:** кастомный протокол — несовместимость; FastAPI — тяжесть; Flask — синхронный.
**Аутентификация (D3 закрыт):** bearer-токен `MCP_AUTH_TOKEN` ДО firewall (`server.py:371-386`).
**Туннель/обнаружение ([D11]/[D12]/[D13]):** облачный Claude достижим только через публичный HTTPS-туннель одной командой; сверка Streamable-HTTP — в S3.
**Где:** `server.py`, `core/transport/transport.py`.

## G11. Поставщик туннеля — Cloudflare Tunnel (cloudflared)
**Q:** через что дать облачному Claude публичный HTTPS-доступ к серверу за NAT одной командой?
**A:** **Cloudflare Tunnel (`cloudflared`)** — исходящее соединение (порты не открывать), HTTPS-терминация у Cloudflare, стабильный именованный URL, один бинарь с keepalive, бесплатно. Форвардит `https://…` → `127.0.0.1:8080`.
**Alt:** `ngrok` (случайный URL на free → ломает коннектор Claude); self-hosted `frp`/`bore` (нужен свой VPS+TLS).
**Регрессия:** смена поставщика меняет только слой запуска (`install.sh`/`run.sh`/`tunnel.py`); quick-tunnel вернёт нестабильный URL. **Один IP туннеля — корень [G12]/[D14].**
**Где:** `install.sh`, `core/transport/tunnel.py`+`run.sh`, `server.py` (bind 127.0.0.1), `config/tunnel.yaml`. Закрывает [D11], стыкуется с [D12]; **не** заменяет аутентификацию [D3].

## G12. Эфемерное in-process состояние файрвола
**Q:** где и как долго живёт защитное состояние (баны, счётчики, история вызовов)?
**A:** ВСЁ состояние firewall — обычные Python-структуры в памяти процесса, без персистентности и без шардинга между воркерами:
- `rate_limiter._requests` (таймстемпы/IP) + `._violations` (счётчик/IP);
- `ip_blocklist._blocked` (dict IP→BlockedIP с TTL);
- `anomaly_detector._method_history` (dict IP→list вызовов).
Дёшево, без внешних зависимостей, достаточно для одного процесса за туннелем.
**Alt:** внешнее хранилище (Redis/SQLite) — отброшено на текущей стадии как избыточное для одного процесса; но именно оно нужно при мультиворкере/рестарт-персистентности.
**Три системных следствия (все доказаны запуском, все — этот кластер):**
1. **Ключ по IP × один IP туннеля** ([G11]) → гранулярность бесполезна: `ip_blocklist.block(tunnel_ip)` режет ВЕСЬ трафик Claude, не клиента ([D14]); anomaly-порог 10 срабатывает на штатной 11-инструментальной сессии ([D17]). Нужна гранулярность по сессии/токену, а не по IP.
2. **Счётчики не сбрасываются/не выселяются** → `rate_limiter._violations` не чистится: после первого бана `unblock` бесполезен, мягкий порог `ban_after` работает лишь ОДИН раз за процесс ([D16]); `anomaly._method_history` не выселяет пустые IP-ключи → memory-DoS при флуде подменных IP ([D19]).
3. **Нет персистентности/allowlist** → рестарт теряет баны; нет постоянного allowlist для IP туннеля, чтобы вывести Claude из-под IP-правил.
**Регрессия:** переход на shared store потребует сериализуемого состояния во всех трёх правилах; добавление воркеров без общего store расщепит защиту (каждый воркер видит свою долю трафика). Фикс гранулярности ([D14]) = новое поле сессии/токена в `FirewallRequest` (см. [G13]).
**Где:** `core/firewall/rules/{rate_limiter,ip_blocklist,anomaly_detector}.py`. Дефекты: [D14], [D16], [D17], [D19]. Файлы: `files/core_firewall_rules_rate_limiter.md`, `..._ip_blocklist.md`, `..._anomaly_detector.md`.

## G13. Контракты вынесены для разрыва цикла импорта
**Q:** как связать модули, которые обмениваются общими типами, без циклического импорта?
**A:** общие модели данных выносятся в **лист-модуль** `contracts`, который сам ничего из своего пакета не импортирует. Применяется дважды и одинаково:
- `core/firewall/contracts.py` — `FirewallDecision/Request/Result`: `rules/*` импортируют из `contracts`, `firewall.py` импортирует и `rules/*`, и `contracts` → цикла нет.
- `core/contracts/*` — `ToolResult/ErrorDetail/Fact/TaskStatus` для всего сервера (тот же приём на уровне ядра; разбор в T2).
**Тип моделей — по назначению:** ядровые контракты (`core/contracts`) на **Pydantic v2** (сериализуются на провод MCP, нужна валидация/схема). Внутренние DTO файрвола — stdlib `@dataclass` (на провод НЕ выходят: `FirewallResult` конвертируется в JSON-RPC-ошибку на краю, `FirewallRequest` собирается из уже-распарсенного тела; лёгкий, без оверхеда на горячем пути). Оба выбора оправданы своим контекстом.
**Alt:** типы в основном модуле — цикл; дублировать в каждом потребителе — рассинхрон.
**Регрессия:** если `contracts`-модуль начнёт импортировать из своего пакета — цикл вернётся (модуль обязан оставаться листом графа); `@dataclass` не валидирует — защита входа держится на гардах края (`server.py` isinstance), при их ослаблении не-типы поедут в правила. Тесно связано с [G6] (порядок определений = граф зависимостей).
**Где:** `core/firewall/contracts.py`, `core/contracts/*`. Файл: `files/core_firewall_contracts.md`. Смежный дефект [D20] (мёртвое поле `error_code` в этом контракте).

## G14. Внутренний `ToolResult` богаче того, что доходит до Claude на проводе
**Q:** почему богатые поля контракта не влияют на поведение Claude?
**A:** между внутренним `ToolResult{status,data,error,facts}` и MCP-`CallToolResult{content,isError,structuredContent,_meta}` стоит маппинг в транспорте — и он кладёт неспецификационные поля мимо `content`, так что спек-совместимый клиент их роняет. Это **зонтик** над кластером находок (все доказаны запуском):
- `facts` → в нестандартном ключе, не в `content` → Claude не видит «память о действиях» ([D24]/[D30]);
- структурный `error` (код/recovery/`raw_response`) → в нестандартном ключе `error`, спек-клиент видит лишь `content[0].text=message` → теряются машинный `error_code` ([D20]), реакционный `class` ([D27]);
- согласованность `status`↔`error`/`result` не форсится ни в `ToolResult`, ни в `TaskStatus` ([D22]).
**Alt:** «раз поле есть в контракте — значит доходит» — эмпирически неверно; граница провода — отдельный слой.
**Регрессия / направление фикса:** единый маппер (`transport._handle_tools_call`) — `facts`/структурный `error` → `structuredContent`; один фикс закрывает невидимость сразу D20/D24/D27. Пока не закрыто — обещания «facts=память», «код→класс» декоративны.
**Где:** `core/transport/transport.py:135-149`; корни — `core/contracts/*`, `core/reactions`. Дефекты: [D30] (точка схода), [D20]/[D24]/[D27]/[D22]/[D23].

## G15. Строковые словари без единого реестра дрейфят; `Literal`/реестр — лекарство
**Q:** почему одни «словарные» поля рассинхронизируются с реальностью, а другие нет?
**A:** там, где значение — свободная `str` без единого источника истины, набор дрейфует; там, где `Literal`/реестр — нет. Доказано:
- **дрейф:** `ErrorDetail.code: str` не привязан к `server_reactions.yaml` → `server.py` эмитит незарегистрированные `PATH_NOT_FOUND`/`FILE_NOT_FOUND`/`TABLE_NOT_FOUND` ([D4]); `Fact.type: str` — эмитируемые типы и словарь docstring имеют НУЛЕВОЕ пересечение ([D25]).
- **эталон (без дрейфа):** `TaskStatus.status: Literal[...]` (невалидное отвергается), `Engine`/провайдеры эмитят только registry-валидные коды.
**Alt:** «дисциплина разработчика удержит словарь» — не удержала (D4/D25 доказаны).
**Регрессия / направление фикса:** генерировать `Literal`/валидатор из единого реестра (коды — из `server_reactions.yaml`; типы фактов — из общего перечня); тест-страж «каждое эмитируемое значение ∈ реестра».
**Где:** `core/contracts/{error_detail,fact}.py`, `server.py`, `core/reactions`. Дефекты: [D4], [D25]; эталон — [core_contracts_task_status.md], [core_engine_engine.md].

## G16. Незавершённость должна КРИЧАТЬ (честная заглушка), а не молчать
**Q:** как отличить «ещё не реализовано» от «сломано/врёт»?
**A:** незавершённая фича обязана явно сигналить (`raise NotImplementedError` с текстом), а не тихо ничего не делать и не заявлять себя готовой. Аудит выявил оба полюса:
- **честно (ок):** `providers/*` и `TaskStatus`-producer'ы → `NotImplementedError("…будет реализован…")` — контракт-first, код «кричит».
- **молча/ложно (дефект):** `facts→_SESSION_LOG` задокументирован, но `log_event` имеет 0 вызовов ([D24]); история v2.0 ЗАЯВИЛА bearer-auth закрытым, а кода не было ([D3] — **закрыт в S6**, но история была антиобразцом честности); `IDGenerator` создаётся, но не вызывается ([D28]).
**Контраст честности доков:** `history_core_reactions.md` (см. SESSIONS.md §Сессия 2) честно помечает D4 незавершённым — образец; `history_server.md` (SESSIONS.md §Сессия 5) про D3 — антиобразец (враньё → ложное чувство защищённости). D3 закрыт (S6), но урок [G8] остаётся.
**Alt:** «доки-нарратив = факт» — ритуал даёт связность, но не проверяемость ([G8]); отсюда v2-бар: доказывать запуском.
**Где:** весь проект. Дефекты: [D24]/[D28] (D3 закрыт); эталон — `core/providers/*`, [core_contracts_task_status.md].

## G17. Containment `workspace/` — единая точка, а не проверка в каждом хендлере

**Q:** где должна жить проверка «путь не выходит за `workspace/`»?
**A:** в ОДНОМ общем месте (choke-point), через которое проходят все ФС-операции — а не копией в каждом входе. **Реализовано (D29 закрыт, S7):** `core/paths.py` → `safe_resolve(path, workspace)`. И `server.py` (fs_*), и `state_manager` (5 методов) зовут его; нарушение → `ValueError` → `PATH_ESCAPE` (registry-код, [G15]). Паттерн: MCP servers filesystem `Path.resolve()` → `is_relative_to(root)`.
**Alt:** «каждый хендлер сам делает safe-join» — отброшено: пропуск (D29) неизбежен; две копии safe-join дрейфуют.
**Регрессия:** если `core/paths.py` начнёт импортировать из своего пакета — цикл; если убрать проверку — traversal вернётся.
**Где:** `core/paths.py` (единая точка), `server.py` (fs_* + json_read_snapshot), `core/state/state_manager.py` (5 методов). **D1 закрыт, D29 закрыт.** Связь: [G9](#g9-две-вселенные-код-сервера-vs-workspace-данные).

## G18. За туннелем клиент один — гранулярность по IP бессмысленна, секрет уязвим

**Q:** что следует из того, что весь трафик Claude идёт через один cloudflared-edge?
**A:** два системных следствия: (1) **один IP** → IP-firewall/rate-по-IP либо бесполезны, либо банят Claude целиком ([D14]/[G12]); нужна гранулярность по сессии/токену, а не IP. (2) **аутентификация обязана быть на уровне приложения**, т.к. туннель — только транспорт, публичный URL — не барьер ([D3] закрыт — bearer `MCP_AUTH_TOKEN` ДО firewall, `server.py:371-386`). Плюс гигиена секрета туннеля: токен в argv/`tunnel.yaml` ([D31]).
**Alt:** «firewall по IP защитит» — отброшено: за туннелем IP не различает клиентов.
**Регрессия / направление фикса:** D3 закрыт (bearer auth); осталось: идентичность по сессии/токену в `FirewallRequest` (D14) + env токена туннеля (D31).
**Где:** `core/transport/tunnel.py` (корень single-IP), `server.py` (auth-слой), `core/firewall/*`. Дефекты: [D14]/[D31] (D3 закрыт). Связь: [G11](#g11-поставщик-туннеля--cloudflare-tunnel-cloudflared), [G12](#g12-эфемерное-in-process-состояние-файрвola).
