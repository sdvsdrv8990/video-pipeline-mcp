# Q&A: core/ids/id_generator.py

> **Роль:** генератор уникальных ID сущностей сервером (не Claude) — `generate(entity_type)→"PREFIX_<uuid4.hex>"`. Реестр префиксов по типам (video/scene/render/task/fact…).
> **Сквозное:** [G10](../global.md#g10-id-генерирует-сервер-не-claude) (осн. — ID генерирует сервер, единый формат); [G6](../global.md#g6-порядок-кода-в-файле--граф-зависимостей-снизу-вверх) (префиксы→методы).
> **Статус кода:** реализован. **D9 закрыт** (полный uuid4 вместо md5[:8] — доказано). НО подсистема пока не вызывается ни одним хендлером (unexercised, ⚪ D28); обещанный `id_registry` не существует.
> **Навигация (знать не читая):** `core/ids/id_generator.py`. Поверхность: `IDGenerator(prefixes=None)` — `generate(entity_type)`, `generate_simple(prefix)`, `get_prefix`, `is_valid_format(id)`; классовый `PREFIXES` (14 типов); приватный `_generate_unique()→uuid4().hex`. Создаётся `server.py:103`, передаётся в `register_basic_tools` — но handlers его НЕ зовут.
> **Аудит-линзы:** mcp-developer (осн. — формат/проводка), test-master (коллизии/формат). Находки доказаны запуском на `.venv`.

## Решение 1: формат `PREFIX_<uuid4.hex>` — сервер владеет ID (G10)
**Q:** кто и как присваивает ID сущностям?
**A:** сервер через `IDGenerator`; формат `PREFIX_xxxx` (`VID_…`, `S_…`). Централизация → единый формат, Claude не выдумывает ID ([G10]). Префиксы читаемы по типу.
**Alt:** ID от Claude — отброшено (коллизии, неконтролируемый формат); голый UUID без префикса — теряется тип сущности.
**Регрессия:** смена формата ломает `is_valid_format` и все ссылки по ID.
**Связь:** [G10](../global.md#g10-id-генерирует-сервер-не-claude); потребитель по замыслу — `Fact.id` ([core_contracts_fact.md]), но сейчас не питается отсюда ([D25]).

## Решение 2: полный `uuid4().hex`, а не `md5[:8]` — D9 закрыт (доказано)
**Q:** сколько энтропии в уникальной части?
**A:** полный `uuid4().hex` (32 hex, 122 бита). Раньше усекался до 8 hex (32 бита) → по парадоксу дней рождения ~50% коллизий на ~77k ID при обещанном, но не реализованном `id_registry`.
**Alt:** `nanoid` заданной длины (эквивалент, короче) — история предлагает как компромисс читаемости; выбран stdlib `uuid4` без зависимости.
**Регрессия / доказано запуском:** `generate('video')→VID_286e1342…b8` (32 hex, `is_valid_format`=True); 50000 ID → **0 коллизий**; старый `VID_abc12345` (8 hex) → `is_valid_format`=**False** (регресс-страж). **Напряжение (не дефект):** фикс противоречит v1-Решению 1 («UUID длинный/нечитаемый») — теперь ID именно длинные. История сама флагует гипотезу читаемости → кандидат на `nanoid`.
**Связь:** [D9 закрыт](../AUDIT.md); SESSIONS.md §Сессия 2 (history_core_ids v2.0).

## Решение 3: `is_valid_format` — страж формата (32 hex lowercase)
**Q:** как отличить валидный ID?
**A:** `split("_",1)` → непустой префикс + `unique` длиной 32 из `[0-9a-f]`. Отвергает старый 8-hex и мусор.
**Alt:** regex — эквивалент; выбран explicit-check.
**Регрессия:** `uuid4().hex` всегда lowercase → согласовано; ID с несколькими `_` → `split(_,1)` берёт хвост целиком (в нём `_` не hex → False, корректно).
**Связь:** [G10](../global.md#g10-id-генерирует-сервер-не-claude).

## Открытые вопросы файла
- **✅ D9 (коллизии 32-бит) — закрыт:** полный uuid4, доказано (0 коллизий на 50k, старый формат отвергается). См. [../AUDIT.md](../AUDIT.md) (Дельта).
- **⚪ D28 (подсистема готова, но не задействована + гигиена):** `IDGenerator` создаётся и передаётся в `register_basic_tools`, но **ни один хендлер не зовёт `.generate()`** (dead injection-параметр); `id_registry` (реестр связей, обещан G10/docstring) **не существует** (grep=0); мёртвые импорты `hashlib`/`time`/`Optional` (остаток md5+timestamp). Инфра впереди использования (текущие 4 tools сущностей не создают) — не поломка, но неупражнённая. См. [../AUDIT.md#-d28](../AUDIT.md#-d28).
- **🟡 связь D25 (`Fact.id` мёртв):** факты, создающие сущность, должны брать ID отсюда — сейчас `id` суют в `data`, `IDGenerator` в обход. Общий фикс с [D25](../AUDIT.md#-d25).
