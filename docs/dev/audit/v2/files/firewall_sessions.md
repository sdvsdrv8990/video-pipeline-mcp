# Q&A: развитие файрвола — план сессий

> **Роль:** план развития файрвола (F1–F8): что сделано, что планируется, ограничения detection.
> **Сквозное:** [G3](../global.md#g3-firewall-перед-ядром), [G12](../global.md#g12-эфемерное-in-process-состояние-файрвола), [G17](../global.md#g17-containment-workspace--единая-точка-а-не-проверка-в-каждом-хендлере), [G18](../global.md#g18-за-туннелем-клиент-один--гранулярность-по-ip-бессмысленна-секрет-уязвим).
> **Статус:** F1 выполнена (базовая реализация), F8 зафиксирована (ограничения). F2–F7 планируются.
> **Навигация:** `core/firewall/` → `config/firewall.yaml` → `audit/v2/files/core_firewall_*.md`.

## F1: Базовая реализация ✅
**Статус:** ВЫПОЛНЕНО (2026-07-01).
**Что создано:** `core/firewall/` (8 файлов), `config/firewall.yaml`, 4 правила: rate_limiter, injection_detector, ip_blocklist, anomaly_detector.
**Решения:**
1. Файрвол ПЕРЕД ядром — фильтрация до обработки
2. Модульные правила — каждый файл = одно правило
3. Автоматический бан — при обнаружении атаки (пересмотрен: бан только для rate-limit)
**Связь:** SESSIONS.md §Сессия 1, [G3](../global.md#g3-firewall-перед-ядром).

## F2: Углублённая детекция injection 🔶
**Статус:** ПЛАНИРУЕТСЯ.
**Цель:** обфускация (Base64, Unicode), multi-turn, indirect injection.
**Ограничения текущего детектора (D7):**
- `lower()` + `in` без границ слова → FP на «act as a narrator» (легитимная озвучка)
- Base64/unicode/пробелы обходят → FN
- Проект — видео-/TTS-пайплайн; доменные фразы = FP
**Альтернатива:** честно пометить как «сигнал, не барьер». Ориентир: OWASP LLM01, protectai/llm-guard.
**Связь:** D7, SESSIONS.md §Приложение Д (firewall v3.0).

## F3: Rate limiting 2.0 🔶
**Статус:** ПЛАНИРУЕТСЯ.
**Цель:** адаптивный rate limiting с учётом поведения.
**Проблемы текущего (D14/D16/D17):**
- Ключ по IP × один IP туннеля = бесполезен
- `_violations` не сбрасывается → мягкий порог одноразовый
- Anomaly порог низкий для сессии
**Связь:** D14, D16, D17, [G12](../global.md#g12-эфемерное-in-process-состояние-файрвола).

## F8: Ограничения detection — XSS и prototype pollution ✅
**Статус:** ЗАФИКСИРОВАНО.
**Что выяснено:**
- InjectionDetector НЕ ловит: XSS (`<script>`), prototype pollution (`__proto__`)
- Ловит: shell injection (`rm -rf`), `delete all`, `drop table`
- Это ограничение, не баг: детектор заточен на prompt injection и shell-команды
**Решение:** whitelist разрешённых типов файлов на уровне engine: `.py`, `.yaml`, `.json`, `.xlsx`. CSS/JS/HTML невозможны по design.
**Гипотезы:**
1. Ограничение типов на уровне engine надёжнее detection паттернов
2. XSS не опасен для сервера (нет рендеринга HTML)
3. Prototype pollution не опасен в Python (`json.loads` безопасен)
**Связь:** D7, SESSIONS.md §Приложение Д, SESSIONS.md §Сессия 2.
