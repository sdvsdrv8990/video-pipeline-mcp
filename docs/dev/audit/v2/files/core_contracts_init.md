# Q&A: core/contracts/__init__.py

> **Роль:** единая точка входа пакета контрактов — `from core.contracts import ToolResult, ErrorDetail, Recovery, Fact, TaskStatus`. Чистый ре-экспорт, без логики.
> **Сквозное:** [G6](../global.md#g6-порядок-кода-в-файле--граф-зависимостей-снизу-вверх) (осн. — порядок импортов = граф зависимостей); [G13](../global.md#g13-контракты-вынесены-для-разрыва-цикла-импорта) (пакет контрактов как единый словарь ядра).
> **Статус кода:** реализован, **чистый** — `__all__` == импорты == реальные классы (дрейфа нет, в отличие от [D25]/[D4]). Дефектов не обнаружено.
> **Навигация (знать не читая):** `core/contracts/__init__.py`. Экспортирует 5 классов из 4 модулей: `Fact` (fact.py), `Recovery`+`ErrorDetail` (error_detail.py), `ToolResult` (tool_result.py), `TaskStatus` (task_status.py). Ничего не вычисляет.
> **Аудит-линзы:** mcp-developer (осн. — фасад пакета). Проверено grep-сверкой на `.venv`.

## Решение 1: единая точка входа = фасад пакета контрактов
**Q:** как импортировать контракты из любого места сервера?
**A:** `from core.contracts import ...` — один фасад вместо путей к подмодулям. Потребитель не знает, в каком файле лежит `ErrorDetail`.
**Alt:** импортировать напрямую `from core.contracts.error_detail import ErrorDetail` — рабоче, но привязывает к раскладке файлов (рефактор модулей ломает импорты).
**Регрессия:** удаление/переименование экспорта ломает импорты по всему проекту (единственный реальный риск этого файла).
**Связь:** потребители — `engine`, `transport`, `reactions`, `server.py`, `providers/*` (все `from core.contracts import …`).

## Решение 2: порядок импортов = граф зависимостей (G6), но безопасность — в подмодулях
**Q:** почему импорты идут `fact → error_detail → tool_result → task_status`?
**A:** снизу вверх по зависимостям ([G6]): `Fact`/`Recovery` — листья, `ToolResult` использует `ErrorDetail`+`Fact`, `TaskStatus` использует `ErrorDetail`. Читается как нарратив «было базовое → появилось использующее».
**Alt:** алфавит/произвольно — отброшено (нить зависимостей теряется).
**Регрессия / нюанс (проверено):** порядок в `__init__` — **документальный**, не функциональный: цикло-безопасность обеспечена тем, что подмодули импортируют соседей НАПРЯМУЮ (`tool_result.py`: `from .error_detail import ErrorDetail`, `from .fact import Fact`), а не через пакет. Тот же приём разрыва цикла, что [G13]. Перестановка строк здесь ничего не сломает — но нарушит читаемость-как-историю.
**Связь:** [G6](../global.md#g6-порядок-кода-в-файле--граф-зависимостей-снизу-вверх) (этот файл — канонический пример), [G13](../global.md#g13-контракты-вынесены-для-разрыва-цикла-импорта).

## Открытые вопросы файла
- **Дефектов нет.** `__all__` (Fact, Recovery, ErrorDetail, ToolResult, TaskStatus) в точности совпадает с импортами и с реальными классами модулей — ни пропущенного, ни лишнего экспорта. Позитивный контраст с дрейфом [D25] (`Fact.type`)/[D4] (`ErrorDetail.code`): здесь словарь экспортов синхронен с кодом.
- **⚪ мелочь (не дефект):** docstring зовёт `Fact` «базовым, используется другими» — фактически `Fact` использует только `ToolResult` (лист, как `Recovery`). Неточность формулировки, порядок при этом корректен.
