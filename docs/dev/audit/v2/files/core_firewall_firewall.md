# Q&A: core/firewall/firewall.py

> **Роль:** фасад файрвола — прогоняет запрос через правила в фиксированном порядке ДО ядра.
> **Сквозное:** [G3](../global.md#g3-firewall-перед-ядром) (firewall перед ядром), [G8](../global.md#g8-дисциплина-документирования) (метрики → проверка гипотез).
> **Статус кода:** реализован; конфиг **подключён** (D2 закрыт), все 4 правила проведены (D8 закрыт).
> **Аудит-линзы:** `security-reviewer` (осн.), `mcp-developer` (контракты/проводка), `test-master` (регрессии). Находки доказаны запуском на .venv Python.

## Решение 1: фиксированный порядок проверок (дёшево→дорого)
**Q:** в каком порядке применять правила?
**A:** IP-blocklist → rate limit → injection → anomaly (`check()`). Дешёвые отсекают раньше, дорогие не тратятся на заведомо забаненных.
**Alt:** произвольный/параллельный порядок — отброшено: лишняя работа на забаненных IP.
**Регрессия:** перестановка меняет и стоимость, и семантику (injection до бана).
**Связь:** правила — `rules/*.py`; типы — `contracts.py`.

## Решение 2: фасад владеет правилами, правила независимы
**Q:** как связать правила и фасад без циклов импорта?
**A:** `Firewall` инстанцирует 4 правила; общие типы (`FirewallRequest/Result/Decision`) в `contracts.py` — разрыв цикла. Strategy-паттерн.
**Alt:** монолитный firewall — отброшено: не тестировать/не поддерживать по частям.
**Регрессия:** новое правило = поле + вызов в `check()`; забыть вызвать → правило мёртвое (историч. случай anomaly, теперь проведён).
**Связь:** [core_firewall_contracts.md](core_firewall_contracts.md).

## Решение 3: реакция на атаку — дифференцированная (уточнено против v1)
**Q:** банить IP при любой сработке?
**A:** нет. **Rate-limit** → мягкий 429, бан IP только после `should_ban()` (порог `ban_after`) — за туннелем Claude один IP, всплеск не должен ронять его на 24ч. **Injection** → блок только текущего запроса, **без** авто-бана (эвристика даёт ложняки, Claude — доверенная сторона).
**Alt:** мгновенный бан при первом превышении (старое поведение, D6/D7) — отброшено как операционная мина за туннелем.
**Регрессия:** если вернуть авто-бан на injection — один ложняк («act as a narrator») забанит Claude целиком.
**Связь:** старое D6 (мгновенный бан) и авто-бан на injection → закрыты здесь; сам детектор с FP/FN → **[D7](../AUDIT.md#-d7)**.

## Решение 4: anomaly конфигурируется из firewall.yaml (D8/D2)
**Q:** откуда `AnomalyDetector` берёт параметры окна?
**A:** из `config["anomaly_detection"]` (`window_sec`, `max_methods_per_window`). Раньше создавался без аргументов → yaml игнорировался и правило было полу-мёртвым.
**Alt:** хардкод окна — отброшено: не настроить под нагрузку без правки кода.
**Регрессия:** рассинхрон ключей `firewall.yaml` ↔ `.get(...)` → тихий откат к дефолтам (проверять тестом схемы конфига).
**Связь:** загрузка — `server.py:96,102`; см. [D2 закрыт](../AUDIT.md#-d2).

## Решение 5: `get_stats()` — счётчики для наблюдаемости
**Q:** как измерять эффективность защиты?
**A:** агрегируем blocked_ips/violations/injection/anomaly — основа для проверки гипотез (G8).
**Alt:** без метрик — гипотезы «снизит на N%» непроверяемы.
**Регрессия:** — ; метрики пока никуда не выводятся (нет харнесса/эндпоинта).

## Открытые вопросы файла
- **🟡 D7 (injection: FP на легитимном + обход):** `detect("act as a narrator")==True`, `detect("disregard the previous take")==True` (FP), а base64/unicode/пробелы → False (FN). Доказано запуском. Детально — в [core_firewall_rules_injection_detector.md](core_firewall_rules_injection_detector.md). См. [../AUDIT.md#-d7](../AUDIT.md#-d7).
- **🟠 D1 (path traversal, остаётся на уровне инструментов):** firewall его НЕ ловит — `detect({"path":"../../../../etc/passwd"}) == False` (доказано). Защита обязана быть в `fs_*` safe-join, не в firewall. См. [../AUDIT.md#-d1](../AUDIT.md#-d1) и [core_firewall_rules_injection_detector.md](core_firewall_rules_injection_detector.md).
- **🟡 D14 (IP-гранулярность за туннелем):** `request.ip` за cloudflared — общий IP; IP-blocklist/rate-по-IP почти бесполезны против Claude-трафика. Нужна гранулярность по сессии/токену, а не по IP. См. [../AUDIT.md#-d14](../AUDIT.md#-d14).
- **✅ D10 (fail-open у вызывающего) — ЗАКРЫТО (верифицировано):** `server.py:359-380` fail-**closed** — исключение вокруг `firewall.check()` → HTTP 403 block (не `except: pass`), нераспарсенное тело → 400. См. [../AUDIT.md#-d10](../AUDIT.md#-d10).
