# Q&A: core/contracts/fact.py

> **Роль:** факт о выполненном действии сервера — `Fact{type,data,id}`. Копится в `ToolResult.facts`, задуман как «память оркестратора»: Claude помнит, что уже сделано.
> **Сквозное:** [G2](../global.md#g2-единый-конверт-ответа-toolresult) (facts — часть единого конверта); [G1](../global.md#g1-роль-сервера-исполнительвалидатор-оркестратор--снаружи) (память держит оркестратор-Claude, не сервер); [G10](../global.md#g10-id-генерирует-сервер-не-claude) (`id` — от `IDGenerator`).
> **Статус кода:** реализован (Pydantic v2, 3 поля). Механически исправен, но задокументированный поток `facts→_SESSION_LOG` НЕ проведён (D24), `id` мёртв + `type` дрейфит (D25).
> **Навигация (знать не читая):** `core/contracts/fact.py`. Поверхность: `Fact(BaseModel)` — `type: str`, `data: dict`, `id: str|None=None`. Производители: `server.py` (DirectoryTree/FileRead/FileCreated/SnapshotRead/Echo), `providers/ffmpeg` (RenderCompleted). Потребитель: `transport.py:142` (`facts:[f.model_dump()…]` — вне MCP-`content`). `_SESSION_LOG` — `state_manager.log_event` (без вызовов).
> **Аудит-линзы:** mcp-developer (осн. — контракт/поток), test-master (регрессии/дрейф). Находки доказаны запуском/grep-проводкой на `.venv`.

## Решение 1: три поля `type` + свободный `data` + опциональный `id`
**Q:** какой минимум описывает «что сделано»?
**A:** `type` (что за действие) + `data: dict` (детали, произвольно под тип) + `id` (присвоенный сущности ID, если есть). Порядок полей — [G6] (тип→содержимое→связь).
**Alt:** типизировать `data` под каждый тип факта — отброшено (на старте): свободный dict гибче, пока набор типов не устоялся.
**Регрессия:** смена формата `data` «сломает парсинг facts в _SESSION_LOG» (из docstring) — но парсинга нет ([D24]); реальный потребитель — только `transport.model_dump()`.
**Связь:** конверт — [core_contracts_tool_result.md · Решение 1] (`facts: list[Fact]=[]`, доказано безопасным).

## Решение 2: facts = память оркестратора (G1), доставка через `ToolResult`
**Q:** зачем возвращать факты, если Claude и так видит результат?
**A:** факты — компактная «память о действиях»: Claude ссылается на них в следующих вызовах, не перечитывая состояние. Память держит оркестратор ([G1]), сервер лишь фиксирует. Замысел — накопление в `_SESSION_LOG` для анализа паттернов через 6 мес.
**Alt:** не возвращать facts, Claude сам трекает — отброшено: сервер знает точные ID/пути, надёжнее отдать факт.
**Регрессия / находка (доказано grep-проводкой):** долгосрочный замысел не подкреплён — `state_manager.log_event` (единственный писатель `_SESSION_LOG.md`) **не вызывается ниоткуда**, и НИ один путь не route'ит `ToolResult.facts` в лог. Факты живут только в проводе; серверной истории нет. См. [D24](../AUDIT.md#-d24).
**Связь:** [G1](../global.md#g1-роль-сервера-исполнительвалидатор-оркестратор--снаружи); транспорт кладёт facts ВНЕ MCP-`content` → спек-клиент может их не показать (тот же parity-разрыв, что [D22]/[D12]).

## Решение 3: `id` опционален — «не все действия присваивают ID» (связь G10)
**Q:** почему `id: str|None`, а не обязательный?
**A:** часть действий создаёт именованную сущность (получает ID от `IDGenerator`, [G10]), часть — нет (чтение, echo). Опциональность отражает это.
**Alt:** обязательный `id` — отброшено: у чтения/echo нет сущности для ID.
**Регрессия / находка (доказано grep):** поле **мёртвое** — `Fact(...)` нигде не задаёт `id=` (все факты `id=None`); идентификаторы вместо этого суют в `data` (`video_id`,`scene_id`). Разрыв с [G10]: выделенное поле под серверный ID не используется, ID живут ad-hoc в свободном `data`. См. [D25](../AUDIT.md#-d25).
**Связь:** [G10](../global.md#g10-id-генерирует-сервер-не-claude), [core_ids_id_generator.md].
