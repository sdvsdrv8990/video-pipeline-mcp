# Q&A: FFmpeg Adapter — единый интерфейс

> **Роль:** адаптер сводит несколько обёрток FFmpeg в единый интерфейс наружу.
> **Сквозное:** [G1](../global.md#g1-роль-сервера-исолнительвалидатор-оркестратор--снаружи), [G16](../global.md#g16-незавершённость-должна-кричать-честная-заглушка-а-не-молчать).
> **Статус:** архитектура описана; обёртки определяются при подборе реп.
> **Навигация:** `core/providers/ffmpeg/ffmpeg_adapter.py` → `wrappers/`.

## Принцип
- Claude видит один набор инструментов монтажа; чем исполнено внутри — деталь адаптера
- Несколько обёрток на бэкенде — норма; несколько обёрток в лице ИИ — хаос
- Выбор обёртки — по надёжности и тестам, не по числу команд

## Архитектура
```
core/providers/ffmpeg/
   ├── wrappers/
   │   ├── wrapper_a/     # обёртка под конкретные операции
   │   └── wrapper_b/     # другая обёртка
   └── ffmpeg_adapter.py  # единый интерфейс наружу
```

## Инструменты (через адаптер)
| Инструмент | Фаза | Назначение |
|---|---|---|
| `trigger_render` | 1 | Запуск рендера |
| `poll_render_status` | 2 | Проверка статуса |
| `download_rendered` | 3 | Скачивание результата |
| `cancel_render` | — | Отмена задачи |

## Сбои монтажа
FFmpeg — локальный движок. Сбой → `LOCAL_INFERENCE_FAILED`-подобный (не PROVIDER_FAILED, не CONTENT_REJECTED). Битый исходник, нет файла, ошибка кодека, нехватка ресурса.

## Гипотезы
- Claude будет использовать `poll_render_status` для длинных рендеров
- `cancel_render` будет использоваться редко (<5% случаев)
- Неверный URL внешнего MCP → все render-операции сломаются
