# Q&A: core/firewall/rules/rate_limiter.py

> **Роль:** ограничение числа запросов с одного IP в скользящем окне; шаг 2 в `Firewall.check()`.
> **Сквозное:** [G3](../global.md#g3-firewall-перед-ядром); [G12](../global.md#g12-эфемерное-in-process-состояние-файрвола) (`_requests`/`_violations` в памяти, ключ по IP — корень D14/D16).
> **Статус кода:** реализован; D6 (замысел `ban_after`) закрыт фасадом, но всплыл новый баг сброса (D16).
> **Навигация (знать не читая):** `core/firewall/rules/rate_limiter.py`. Поверхность: `check(ip,timestamp)->RateLimiterResult`, `should_ban(ip)->bool`, `get_violations()`, `get_ip_violations(ip)`. Вызывается из `firewall.py` шаг 2; `should_ban` — там же перед `ip_blocklist.block()`. Состояние: `_requests` (list таймстемпов на IP), `_violations` (счётчик на IP).
> **Аудит-линзы:** security-reviewer (осн.), test-master (регрессии). Доказано запуском на `.venv`.

## Решение 1: скользящее окно на списке таймстемпов
**Q:** как считать частоту за окно?
**A:** на IP — список времён; `_cleanup` отбрасывает старше `window_sec`; сравниваем длину с `max_requests`.
**Alt:** фиксированные бакеты по минутам — проще, но всплески на границе бакета; выбрали скользящее.
**Поток данных:** `check → _cleanup(ip) → len(_requests[ip]) ≥ max? → блок/добавить таймстемп`.
**Регрессия / где проявится:** список растёт O(запросов в окне) на IP до чистки; при флуде — память и стоимость обхода на горячем пути перед ядром.

## Решение 2: при превышении — не добавлять запрос, инкремент violations
**Q:** считать ли отклонённый запрос в окно?
**A:** нет — не добавляем в `_requests`, инкрементим `_violations[ip]`. Отклонённые не удлиняют окно.
**Alt:** добавлять всё — отброшено: искажает окно, растягивает бан.
**Регрессия:** — .

## Решение 3: раздельный учёт нарушений + порог `ban_after` (D6 закрыт, но D16 открыт)
**Q:** как отличить всплеск от бота?
**A:** `_violations[ip]`, `should_ban() = violations ≥ ban_after`. Замысел — банить не с первого раза. **Фасад теперь это использует** (D6 закрыт: `firewall.py` зовёт `should_ban` перед `block`). Доказано: при `ban_after=2` — `should_ban` становится True на 2-м нарушении, не на 1-м.
**Alt:** бан сразу — грубо для легитимных.
**Регрессия / новый дефект → [D16](../AUDIT.md#-d16):** `_violations` **никогда не сбрасывается** (нет reset при бане/разбане/по времени). **Доказано:** бан → `unblock` → `violations` остаются (=4) → **одно** новое превышение → `should_ban=True` мгновенно. Итог: мягкий порог работает лишь **один раз за жизнь процесса**; после первого бана `unblock` бесполезен, любой единичный всплеск = мгновенный ре-бан. **Где проявится:** за туннелем (один IP, [D14](../AUDIT.md#-d14)) первый же разбан Claude ничего не даёт — следующий всплеск снова валит весь трафик. **Фикс:** сбрасывать/декрементить `_violations[ip]` при `unblock` и/или по TTL окна.

## Открытые вопросы файла
- **🟠 D16:** violations без сброса (см. Решение 3).
- **Сквозное (→ global.md кандидат):** состояние в памяти процесса — не переживает рестарт, не шардируется между воркерами (общее с `ip_blocklist`). Кандидат в `G#` «эфемерное in-process состояние firewall».
- **Доработка:** самодельный лимитер дублирует зрелые решения — ориентир `alisaifee/limits` (moving-window/token-bucket) или `slowapi`; развести «429 мягко» и «бан после N».
- **Нет allowlist** для IP туннеля Claude — усиливает D14/D16.
