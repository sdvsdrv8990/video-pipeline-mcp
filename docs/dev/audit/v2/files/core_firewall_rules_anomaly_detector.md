# Q&A: core/firewall/rules/anomaly_detector.py

> **Роль:** детекция подозрительного поведения по истории вызовов IP; шаг 4 (последний) в `Firewall.check()`.
> **Сквозное:** [G3](../global.md#g3-firewall-перед-ядром); [G12](../global.md#g12-эфемерное-in-process-состояние-файрвола) (`_method_history` в памяти, ключ по IP — корень D17/D19; общий с rate_limiter/ip_blocklist).
> **Статус кода:** реализован; D8 закрыт (детектор теперь СРАБАТЫВАЕТ — доказано), но фикс вскрыл FP за туннелем (D17) и хардкод-реестр опасных tools (D18).
> **Навигация (знать не читая):** `core/firewall/rules/anomaly_detector.py`. Поверхность: `check(request)->AnomalyResult`, `get_detected()->int`, `_cleanup(ip,t)`. Вызывается из `firewall.py` шаг 4; при `detected=True` фасад → `FirewallDecision.BLOCK`, `error_code="ANOMALY_DETECTED"` (бан НЕ ставится — в отличие от rate_limiter). Состояние: `_method_history` (dict ip→list[(method,ts)]), `_detected` (счётчик). Порядок полей: пороги → история → методы.
> **Аудит-линзы:** security-reviewer (осн.), mcp-developer (привязка к реестру tools), test-master (регрессии). Доказано запуском на `.venv`.

## Решение 1: история `(method, timestamp)` на IP в скользящем окне
**Q:** как заметить отклонение от нормы поведения, ортогонально частоте?
**A:** копим `(method, ts)` на IP в `_method_history`, чистим `_cleanup` по `window_sec`, считаем число УНИКАЛЬНЫХ методов за окно (сигнал «разнообразие действий», а не «частота» — частоту закрывает rate_limiter).
**Alt:** глобальная норма по всем IP — шумнее и сложнее; выбрали per-IP.
**Поток данных:** `check → append(ip) → _cleanup(ip) → set(методов в окне) → len > порог? → AnomalyResult(detected)`.
**Регрессия / где проявится:** двойная фильтрация окна (`_cleanup` + повторный `if t > timestamp - window_sec` в set-comprehension) — избыточно, но безвредно. `_method_history` растёт O(вызовов в окне) на IP до чистки.

## Решение 2 (фикс D8): анализируем `params.name`, а не транспортный JSON-RPC метод
**Q:** почему в v1 детектор не срабатывал никогда?
**A:** через MCP приходят лишь ~4 транспортных метода (`tools/call`, `tools/list`, `initialize`, `ping`) — при пороге 10 набрать >10 уникальных невозможно. Фикс: для `tools/call` брать `params.name` (имя инструмента) — реальное разнообразие действий несёт оно.
**Alt:** оставить транспортный метод — эмпирически мёртвый детектор (v1 D8).
**Поток данных:** `request.method=="tools/call" & params dict → method = params["name"]; иначе → request.method`.
**Регрессия / где проявится → D8 ЗАКРЫТ (доказано):** `AnomalyDetector(max_methods_per_window=10)`; 12 разных `params.name` через `tools/call` → `detected=True, "Слишком много уникальных методов: 12"`. Контроль: 50× `tools/list` (1 уникальный транспортный метод) → `detected=False`. Детектор действительно ожил. **Но** живой детектор с низким порогом за одним IP туннеля стал источником FP → [D17](../AUDIT.md#-d17).

## Решение 3: чёрный список «опасных» инструментов `DANGEROUS_TOOLS` (severity high)
**Q:** как ловить деструктив независимо от разнообразия?
**A:** `frozenset` имён (`fs_delete`, `fs_remove`, `config_delete`, `system_shutdown`…) → один вызов = `BLOCK` sev=high.
**Alt:** только порог разнообразия — пропускает единичный опасный вызов.
**Поток данных:** `method in DANGEROUS_TOOLS → AnomalyResult(detected=True, severity="high")`.
**Регрессия / новый дефект → [D18](../AUDIT.md#-d18):** список — хардкод-угадайка имён, НЕ связанная с реальным реестром tools (`tools/*` пусты, `config/ops/` отсутствует). **Доказано:** `check(name="fs_delete")` → `BLOCK` с одного вызова. Когда tools приземлятся: легит-инструмент с таким именем окажется вечно недоступен (FP — firewall режет его 100% вызовов), а реальный деструктив под другим именем пройдёт (FN). Имена должны браться из метаданных tools/`config`, а не из хардкода в правиле.

## Решение 4: `AnomalyResult` с severity (`low/med/high`)
**Q:** как ранжировать сигнал?
**A:** `low/med/high` — приоритизация реакции/логов; med для порога разнообразия, high для опасного tool.
**Alt:** boolean — теряется тяжесть.
**Регрессия:** severity нигде не используется фасадом — `firewall.py` ставит `BLOCK` одинаково при любом severity. Пока декоративно.

## Открытые вопросы файла
- **🟡 D17** (см. AUDIT): порог 10 × агрегация под одним IP туннеля → легит-сессия пайплайна = `BLOCK` всего Claude. Смягчает то, что бан НЕ ставится (только текущий запрос), но окно 60с держит блок до старения вызовов.
- **🟡 D18** (см. AUDIT): `DANGEROUS_TOOLS` оторван от реестра tools (FP+FN).
- **⚪ D19** (см. AUDIT): `_method_history` не выселяет IP-ключи (доказано: 256 ключей на 1000 прямых коннектов) — рост памяти при потоке прямых IP (bot_army); за туннелем 1 ключ — не проявляется.
- `_detected` — счётчик статистики, никогда не сбрасывается; для бана не используется (в отличие от `_violations`/D16), потому — Info.
- «Норма определяется автоматически» (docstring, ур. 6мес) НЕ реализовано — пороги статические. Либо убрать обещание, либо ввести baseline-обучение (сессия F6 «планируется»).
- **Сквозное (→ global.md кандидат):** состояние в памяти процесса — не переживает рестарт, не шардируется, нет allowlist для IP туннеля. Общее с `rate_limiter`/`ip_blocklist` → синтезировать в `G#` при проходе S1.
