# Каталог аудита v2 — операционный мануал

Это **самодостаточная инструкция**: по ней ИИ (или человек) с нуля понимает *что делать, что читать, в каком порядке, по каким правилам*. Если ты открыл только этот файл — прочитай раздел **0** и дальше по ссылкам.

Аудит v2 переделывает разбор проекта на базе v1 (`../`), применяя 3 адаптированных скила, сохраняя формат v1, но добавляя доказуемость (эмпирику) и находки «что улучшить». Старый `../` не трогаем, пока v2 не готов.

---

## 0. Холодный старт (с чего начать за 5 минут)

Порядок чтения, чтобы начать работать:
1. **Этот README** — правила и структура (ты здесь).
2. **SESSIONS.md** — хронология: что сделано, что нашли, что решено.
3. **AUDIT.md** — дефекты D# (что улучшить).
4. **questions.md** — максимальный пул инженерных вопросов (`../../questions.md`).
5. **Протокол контекста** — `../../CONTEXT_PROTOCOL.md` (порядок чтения для каждой задачи).
6. **Процесс проекта** — `../../workflow.md` (новый процесс v2). Это авторитет; при расхождении побеждает он.
7. Взять **один вопрос** из questions.md (статус ⬜ или 🔶) → выполнить его → обновить SESSIONS.md/AUDIT.md.

**Железные правила:** один файл за сессию · находки доказывать запуском · не копировать код/тексты в память · без дублей (находка живёт в одном месте).

## 2. Принцип пирамиды (знание слоями, без дублей)

```
          ▲  AUDIT.md (D#)         ← «что улучшить»: дефекты, тонкий верхний слой
         ▲▲  files/*.md (Q&A)      ← «почему так» для каждого файла
        ▲▲▲  global.md (G#)        ← «почему так» сквозное (общая база решений)
       ▲▲▲▲  workflow.md + код     ← фундамент: процесс проекта и исходники (истина)
```

- **Читаем сверху вниз** для ориентации (быстрый верх → детальный низ), **пишем снизу вверх** (сначала решение в файле/`global`, потом дефект в `AUDIT`).
- **Без дублей:** сквозное решение — один раз в `global.md`, файл только ссылается (`[G3](../global.md#g3-firewall-перед-ядром)`). Дефект — один раз в `AUDIT.md` (`D#`), файл ссылается из «Открытых вопросов».
- Разные жанры не пересекаются: `AUDIT.md` = «что не так», `global/files` = «почему так». Находка аудита ссылается на решение (`D7 → files/core_firewall_rules_injection_detector.md · Решение 2`).

## 3. Порядок чтения перед задачей (MD и код)

Зависит от типа задачи (`../../workflow.md` → «Типы задач»). Общий скелет **перед тем как трогать файл**:

1. `../../workflow.md` — освежить процесс (один раз за сессию).
2. `global.md` — сквозные правила `G#`.
3. `files/<нужный>.md` — решения этого файла (если уже есть).
4. `AUDIT.md` — если файл фигурирует в `D#`, учесть перед выводами.
5. Домен-доки: security → `../../threat_landscape.md` + `../../firewall_sessions.md`; тесты → `../../testing_strategy.md`; модуль → `../../history_<модуль>.md`, `../../session_N_*.md`.
6. **Только теперь — исходник кода**, уже зная контекст «почему так».

Обратный порядок при записи: принял решение → Q&A в `files/` (или `global.md`, если сквозное) → при дефекте завёл `D#` в `AUDIT.md`.

## 4. Формат Q&A (обязательный для `files/*`)

```markdown
# Q&A: <путь к файлу>

> **Роль:** одна строка — зачем файл существует.
> **Сквозное:** ссылки на G# в global.md.
> **Статус кода:** реализован / заглушка / частично + краткий флаг проблем.
> **Навигация (знать не читая):** где файл, публичная поверхность (методы), кто вызывает, порядок полей.
> **Аудит-линзы:** какие скилы применены; «находки доказаны запуском».

## Решение N: <короткий заголовок>
**Q:** какой инженерный вопрос стоял?
**A:** что решили и почему.
**Alt:** что рассмотрели и отбросили, почему.
**Поток данных:** куда и как двигаются данные в этом решении.
**Регрессия / где проявится:** что сломается при изменении, ГДЕ это всплывёт в системе, в чём выразится.
**Связь:** ссылки на файлы/решения/D# (без дублирования текста).

## Открытые вопросы файла
- Дефекты (со ссылкой на D#), концептуальные сомнения, что решить позже.

## Что улучшить (линза test-master и др.)
- Регрессия-тесты на каждую находку; заметки по онбордингу и поведению через 6 мес.
```

**Измерения, на которые отвечает каждый разбор** (часть — только после нескольких проходов, но закладываются все):
навигация (насколько просто ориентироваться) · поток данных (куда течёт и почему) · режим отказа (если сломается — почему, где проявится, в чём) · влияние на систему (как решение отразится и где) · онбординг (сложно ли новому разработчику) · горизонт 6 мес (как поведёт себя система).

## 5. Формат дефектов `D#` (в `AUDIT.md`)

```markdown
### <emoji severity> D<N>. <заголовок>
`file:line` — краткая локализация.
**Доказательство (реальный запуск):** вывод команды/скрипта на .venv.
**Почему важно:** влияние + ссылка на threat_landscape.md/спеку.
**Как чинить:** ремедиация с эталоном-референсом (напр. werkzeug.safe_join).
**Связь:** files/... · Решение N.
```

- **Severity:** 🔴 Critical · 🟠 High · 🟡 Medium · ⚪ Low/Info (CVSS в скобках при уместности).
- **Нумерация сквозная:** v1 занял `D1–D13` — **сохраняем их смысл**, новые находки продолжают с `D14`. Не переиспользовать номер под другой смысл.
- **Дельта против v1:** если дефект v1 закрыт — фиксировать в таблице «Дельта» с доказательством; если уточнён — обновлять тот же `D#`.
- **AUDIT.md — живой ХАБ (обновлять после КАЖДОГО прохода, см. §7.1):** не пишется «в конце». Содержит: таблицу **«Покрытие»** (файл → статус ⬜/🔶/✅ = точка входа и карта прогресса), таблицу **«Дельта»** (закрытые/уточнённые `D#`), **реестр `D#`** со ссылками на `files/*`. Должен опережать или не отставать от `files/*` — по нему одному видно, где система и что дальше.

## 7. Воронка и повторные проходы (главный принцип масштаба)

**Один файл за сессию.** Диск (`files/*`, `global.md`, `AUDIT.md`) — это аккумулятор понимания; память Claude — крошечный реестр (без кода); контекст сессии — 1 файл + его разбор + нужные доки.

**Проходы (definition of done):**
- **Проход 1 — draft:** прочитать исходник + историю + прогнать эмпирику → написать `files/<f>.md` (Q&A) + новые `D#`. Статус файла → 🔶.
- **Проход 2 — verify/integrate:** перечитать исходник, воспроизвести эмпирику, сшить с `global.md` (кросс-файловые `G#`), закрыть/повысить открытые вопросы. Статус → ✅ done (стабильная база воронки).

**Зачем:** done-файл переоткрывается редко и через свой `.md`, а не через сырой код → число проходов и токенов падает; растёт прозрачность проекта.

**Статусы:** ⬜ не начат · 🔶 pass1/draft · ✅ done.

### 7.1 Порядок обновления после КАЖДОГО прохода (write-back — чтобы ничего не пылилось)

После завершения прохода по файлу — обновить сверху вниз по пирамиде, **в этом порядке**, за один заход:
1. **`files/<f>.md`** — сам Q&A (первично).
2. **`global.md`** — если проход вскрыл/затронул сквозное решение → создать/обновить `G#` (иначе пропустить).
3. **`AUDIT.md` (ХАБ)** — обязательно, даже если новых `D#` нет: обновить таблицу «Покрытие» (статус файла), таблицу «Дельта» (закрытые/уточнённые `D#`), добавить/поправить `D#` и перекрёстные ссылки. **AUDIT.md не должен отставать от `files/*` ни на один проход.**
4. **Память `audit-v2-task.md`** — реестр + ledger + ОДНО следующее действие.

Правило: проход не считается закрытым, пока не пройдены все 4 шага. Так `AUDIT.md` всегда актуален как точка входа, а ни один файл не «зависает» в рассинхроне.

### 7.2 Определение прохода (ПЕРЕД началом — чтобы проходы были осмысленны, а не для галочки)

Перед любым проходом ответить (коротко, можно в память/в шапку файла):
1. **Сколько и какие файлы** нуждаются в проходе сейчас? (скан статусов: ⬜ → нужен pass1; 🔶 → готов ли к pass2; ✅ → не инвалидирован ли новой сквозной находкой, требующей касания).
2. **Цель и задачи** этого прохода? (draft / verify / интеграция в `global.md` / ре-аудит после фикса).
3. **Чем улучшит и что добавит** проход существующей системе? (новые `D#`, закрытые `D#`, новые `G#`, заполненные измерения из §4).
4. **Как повлияет на пирамиду?** Проход должен двигать знание вверх (код → файл → `global` → `AUDIT`) и держать хаб консистентным; отметить, в какой слой он вносит (база `G#` / середина `files` / верх `D#`).

**Протокол закрытия сессии:** выполнить write-back (7.1) → записать ОДНО следующее действие, чтобы новая сессия стартовала с одного явного шага.

## 8. Эмпирический стандарт

Находка без доказательства = гипотеза (помечать как таковую). Доказательство — запуск на `.venv/bin/python` с реальным выводом (как v1 доказал path traversal на `/etc/passwd`). Для правил firewall — вызывать `detect`/`check` на реальных payload'ах; для контрактов — импорт и сериализация; для проводки — grep вызова + запуск.

## 9. Холодный старт без памяти (восстановление состояния)

Если память Claude недоступна:
1. Прочитать этот README (правила) + `../../workflow.md` (процесс).
2. `ls files/` → сравнить с охватом (раздел 1): чего нет = ⬜ не начато; что есть — открыть и по последнему разделу понять, pass1 или done.
3. Прочитать `AUDIT.md` (текущие `D#` и дельту) — что уже найдено.
4. Взять первый ⬜/🔶 файл по приоритету (security-блок → ядро → providers) и выполнить проход (раздел 7).
5. В конце — восстановить реестр в памяти (`audit-v2-task.md`), см. раздел 0.
