### А. Философия тестирования (session_1 §6)

#### 5 вопросов перед тестом
1. **Что тестируем?** — конкретный сценарий
2. **Зачем нужен этот тест?** — какую проблему решает
3. **Что хотим увидеть?** — ожидаемый результат
4. **Как тест отражает реальное поведение?** — эмуляция реальных действий
5. **Это постоянный тест или быстрый?** — определяет структуру

#### 4 уровня анализа (на каждый файл)
| Уровень | Вопрос |
|---|---|
| Код | Что меняется в конкретном файле? |
| Поведение | Как изменится работа системы? |
| Поток данных | Как двинутся данные между компонентами? |
| Долгосрочный | К чему приведёт через 6 месяцев? |

#### Типы тестов безопасности
| Тип | Что эмулируем |
|---|---|
| Prompt injection | Вредоносные JSON-запросы |
| Cache overflow | Переполнение кеша сервера |
| Bot army | Армия ботов (100+ одновременных запросов) |
| Virus injection | Внедрение вредоносного кода |
| Cache injection | Инъекция в кеш |
| Slowloris | Медленные соединения |

#### Цикл улучшения
```
Нашли bypass на GitHub → Создали тест → Проверили наш сервер
→ Если уязвимы → Исправили → Добавили регрессию
→ Если защищены → Отметили в history.md
```

#### Быстрый vs постоянный тест
- **Быстрый:** `tests/quick/` → создали → проверили → удалили
- **Постоянный:** `tests/<name>/` + `history.md` → живёт в проекте → при изменениях проверяем

### Б. Три природы ошибок провайдеров (детальный flow, session_1 §3 + server_reactions.md)

```
Провайдер → исключение/статус
    ↓
Сервер: match_error_to_reaction(e) → code (из server_reactions.yaml)
    ↓
Сервер: get_recovery(code) → Recovery{suggested_tool, suggested_params, reason}
    ↓
Сервер: ErrorDetail{code, message=ПОЛНЫЙ текст, recovery, raw_response=ПОЛНЫЙ ответ}
    ↓
Claude: читает recovery → решает (retry / переформулировать / сменить / человек)
```

**Три природы:**

| Природа | Код | Что делает сервер | Что делает Claude |
|---|---|---|---|
| Тех.сбой провайдера | `PROVIDER_FAILED` | retry 3 раза → fallback → уведомление | ждёт/смена |
| Контент-отказ | `CONTENT_REJECTED` | нет retry (тот же промпт → тот же отказ) | переформулировать |
| Локальный сбой | `LOCAL_INFERENCE_FAILED` | 1 retry → деградация модели → человек | ждёт |

### В. Провайдеры: гипотезы и архитектура (history_*.md)

#### FFmpeg adapter (history_core_providers_ffmpeg.md)
**Архитектура:** `Claude → наш сервер → ffmpeg_adapter → внешний MCP-сервер → FFmpeg`

| Инструмент | Фаза | Назначение |
|---|---|---|
| `trigger_render` | 1 | Запуск рендера |
| `poll_render_status` | 2 | Проверка статуса |
| `download_rendered` | 3 | Скачивание результата |
| `cancel_render` | — | Отмена задачи |

**Гипотезы:**
- Claude будет использовать `poll_render_status` для длинных рендеров
- `cancel_render` будет использоваться редко (<5% случаев)
- Неверный URL внешнего MCP → все render-операции сломаются

#### TTS (history_core_providers_tts.md)
**Гипотезы:**
- Большинство TTS будут sync (poll вырождается)
- Неверный API URL → все TTS операции сломаются
- Изменение формата ответа LiteLLM → адаптер вернёт ошибку

#### STT (history_core_providers_stt.md)
**Гипотезы:**
- Пословные таймкоды точнее, чем посекундные
- Нет GPU → медленная обработка (fallback на CPU)
- Модель не загружается → все STT операции сломаются

#### IMG (history_core_providers_img.md)
**Гипотезы:**
- Контент-отказ будет в ~10% случаев (ложные срабатывания)
- Переиспользование исходников сэкономит 30-50% запросов
- Одна сцена → НЕСКОЛЬКО запросов `img_*` (диффузия выдаёт строительные блоки)

**Архитектура (единая для всех async провайдеров):**
```
Фаза 1: trigger_* (запуск) → TaskStatus{task_id, status:"pending"}
Фаза 2: poll_* (опрос)    → TaskStatus{status:"pending|completed|failed"}
Фаза 3: download_*        → ToolResult{data:{file_path}, facts:[...]}
```

### Д. Tunnel bug и firewall ограничения

#### Tunnel: Unregistered regex (history_core_transport_tunnel.md v1.2)
**Баг:** `Unregistered tunnel connection` содержал подстроку `registered tunnel connection` → ложно матчился как «соединение установлено».
**Фикс:** проверка `disconnected` ДО `connected` + lookbehind `(?<!un)`.
**Поймано:** оффлайн-тестом на настоящих форматах логов.

#### Firewall v3.0: XSS и prototype pollution (history_core_firewall.md)
**Ограничение:** injection_detector НЕ ловит:
- XSS (`<script>`) — нет рендеринга HTML → не опасно для сервера
- Prototype pollution (`__proto__`) — Python `json.loads` не подвержен

**Ловит:** shell injection (`rm -rf`), `delete all`, `drop table`, prompt injection (`ignore previous`, `you are now`).

**Решение:** whitelist разрешённых типов файлов на уровне engine (не firewall): `.py`, `.yaml`, `.json`, `.xlsx`. CSS/JS/HTML невозможны по design.

### Е. Динамический конфиг провайдеров (session_2 §2.8 + session_4 §4.5)

**Проблема:** API ключи проектозависимые, нельзя хардкодить.

**Решение:** `workspace/<project>/providers.yaml` — каждый проект свой конфиг.

**Формат:**
```yaml
tts:
  provider: openai
  api_key: os.environ/OPENAI_API_KEY
  model: tts-1
  voice: alloy
img:
  provider: openai
  api_key: os.environ/OPENAI_API_KEY
  model: dall-e-3
stt:
  provider: local
  model: base
```

**Порядок чтения:**
1. Провайдер запрашивает конфиг у state_manager
2. state_manager читает `workspace/<current_project>/providers.yaml`
3. Если файла нет → `config/model_routing.yaml` (дефолт)
