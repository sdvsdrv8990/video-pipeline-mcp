# Тесты: структура и инструкции

## Назначение

Тесты проверяют поведение сервера через реальные сценарии, а не проверку кода.

## Структура

```
tests/
├── README.md                    — этот файл (инструкции для ИИ)
├── quick/                       — быстрые тесты (удаляются после проверки)
│   ├── README.md                — правила быстрых тестов
│   ├── test_firewall.py         — интеграция с живым сервером
│   ├── test_audit_fixes.py      — регрессия аудита D1–D13 (постоянный)
│   └── test_tunnel.py           — регрессия туннеля (постоянный)
├── bot_army/                    — армия ботов (Security)
├── cache_injection/             — инъекция в кеш (Security)
├── cache_overflow/              — переполнение кеша (System)
├── config_change/               — изменение конфига (System)
├── virus_injection/             — вирусная инъекция (Security)
├── render_draft_final/          — рендер сцен (Workflow)
├── config/                      — конфигурация тестов
├── filesystem/                  — тесты файловой системы
├── pipeline/                    — тесты пайплайна
└── tables/                      — тесты таблиц
```

## Типы тестов

### 1. Быстрые тесты (`quick/`)
- **Срок жизни:** от одного запуска до удаления
- **Цель:** быстрая проверка гипотезы
- **Правило:** создал → запустил → удалил
- **Исключение:** `test_audit_fixes.py` и `test_tunnel.py` — постоянные регрессии

### 2. Постоянные тесты (остальные папки)
- **Срок жизни:** живут в проекте
- **Цель:** регрессия и безопасность
- **Структура:** `test_<name>.py` + `history.md`
- **Правило:** при изменении правил — проверяем что тест проходит

## Порядок чтения перед работой с тестом

```
1. Читаем README.md (этот файл)
   - Понимаем структуру и типы тестов

2. Читаем history.md теста
   - Что тестируем
   - Зачем нужен
   - Какие решения были приняты

3. Читаем исходник теста (test_*.py)
   - Что конкретно проверяем
   - Какие сценарии

4. Проверяем: не дублируем ли мы что-то
   - Есть ли похожий тест?
   - Неужели это уже решалось?

5. Запускаем и проверяем
```

## 5 вопросов перед тестом

Перед созданием любого теста отвечаем:

1. **Что тестируем?** — конкретный сценарий
2. **Зачем нужен этот тест?** — какую проблему решает
3. **Что хотим увидеть?** — ожидаемый результат
4. **Как тест отражает реальное поведение?** — эмуляция реальных действий
5. **Это постоянный тест или быстрый?** — определяет структуру

## Порядок работы ИИ

### Новый тест

```
1. Определяем тип: быстрый или постоянный
2. Заполняем таблицу планирования (см. ниже)
3. Создаём структуру: tests/<name>/test_<name>.py + history.md
4. Отвечаем на 5 вопросов → записываем в docstring теста
5. Пишем скрипт: минимум кода, только проверка
6. Запускаем и проверяем
7. Принимаем решение: удалить (быстрый) или оставить (постоянный)
```

### Исправление теста

```
1. Читаем history.md теста
   - Ищем похожие ошибки в прошлом
   - Смотрим какие выводы были сделаны

2. Читаем исходник теста
   - Понимаем контекст: что тестируем

3. Определяем причину (root cause analysis)
   - Что именно сломалось?
   - Почему это произошло?

4. Исправляем с учётом истории
   - Если похожая ошибка была — смотрим как чинили тогда

5. Фиксируем в истории теста
   - Версия X.Y: что исправили
   - Причина ошибки: почему произошла
```

## Таблица планирования (для постоянных тестов)

Перед созданием постоянного теста заполняем:

| Тест | Что эмулируем | Ожидаемый результат |
|---|---|---|
| `bot_army` | Массовое подключение ботов | Rate limiting + бан |
| `cache_injection` | Инъекция в кеш | Файрвол блокирует |
| `cache_overflow` | Переполнение кеша | Кеш не повреждается |
| `config_change` | Изменение конфига | Конфиг защищён |
| `virus_injection` | Вирусная инъекция | Файрвол блокирует |
| `render_draft_final` | Рендер сцен | Сцены рендерятся |

## Формат history.md теста

```markdown
# История теста: <name>

## Статус: РАБОЧИЙ ТЕСТ / НЕ РАБОТАЕТ / УСТАРЕЛ

## Назначение
(одно предложение: что проверяем)

## Тип теста
Security / System / Integration / Workflow

## Что тестируем
1. (сценарий 1)
2. (сценарий 2)

## Зачем нужен
1. **Безопасность/Регрессия/Верификация:** (причина)

## Что хотим увидеть
- (ожидаемый результат 1)
- (ожидаемый результат 2)

## Как отражает реальное поведение
(эмуляция реальных действий)

## Workspace
ДА/НЕТ — (описание)

## Версии

### Версия X.Y — ДАТА
- (что сделали)

## Регрессии
- (что сломается при изменении)
```

